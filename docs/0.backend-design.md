# flist 后端技术设计文档

## 1. 项目概述

flist 是一个面向无界面 NAS 的 Web 文件管理器，提供文件在线浏览、查看、复制、粘贴、删除、移动、搜索等功能，并支持目录收藏夹快捷跳转。

- 技术栈：Go（后端）+ Vite（前端 SPA）+ SQLite（元数据存储）
- 目标平台：Linux / Windows 双平台兼容
- 部署形态：单进程二进制，前端构建产物通过 `embed` 打包进二进制，单文件交付

本文档只描述总体设计与关键决策，不涉及完整实现代码。具体函数签名、错误处理细节以代码为准。

## 2. 跨平台兼容设计

### 2.1 核心原则

| 层面 | 策略 |
|------|------|
| API 路径格式 | 统一使用 `/` 分隔，后端在入口处转为 OS 本地路径，出口处转回 |
| 存储路径格式 | 数据库统一存 `/` 分隔的「相对 root 的 API 路径」，与运行平台解耦 |
| 路径处理 | OS 本地路径用 `filepath` 包，API 路径用 `path` 包 |
| 平台差异代码 | 通过 build tags（`_linux.go` / `_windows.go`）隔离，对上层提供统一接口 |

路径转换、磁盘空间、隐藏文件检测、文件属性等平台相关能力，统一封装在 `internal/util` 下，对 service 层暴露平台无关的接口。

### 2.2 平台差异点汇总

| 差异点 | Linux | Windows | 处理策略 |
|--------|-------|---------|----------|
| 路径分隔符 | `/` | `\` | 入口/出口统一转换 |
| 根目录形态 | `/mnt/data` | `D:/share` | 启动时 `filepath.Abs` 标准化 |
| 文件名非法字符 | `/`、`\0` | 额外禁 `<>:"/\|?*` 及保留名（CON/PRN/NUL/COM1-9/LPT1-9），不可以空格或点结尾 | 校验函数按平台分支 |
| 隐藏文件 | `.` 前缀 | `FILE_ATTRIBUTE_HIDDEN` 属性 | 平台特定检测 |
| 长路径 | 无限制 | MAX_PATH=260，启用后达 32767 | 配置项控制是否加 `\\?\` 前缀 |
| 权限模式 | 标准 `rwx` 位 | 无 Unix 权限位 | 见 2.3 |
| 符号链接 | 正常解析 | 创建需管理员权限，解析可能失败 | 见 2.4 |
| 后台运行 | systemd | Windows 服务 / NSSM | 见部署章节 |

### 2.3 文件权限的跨平台表示

API 的 `mode` 字段统一返回八进制字符串（如 `"0644"`）：

- Linux：直接取 `os.FileMode` 权限位，格式化为八进制
- Windows：无 Unix 权限位，做简化映射 —— 目录 `0755`，只读文件 `0444`，可读写文件 `0666`

前端依据 API 返回的 `os` 字段决定如何展示，避免对 Windows 的简化值做过度解读。

### 2.4 符号链接处理

flist 自身不创建符号链接，但列目录时需正确处理已存在的：

- 用 `os.Lstat` 获取条目信息，不跟随链接
- 标记 `is_symlink`，尽力解析目标路径供前端展示
- 链接目标必须仍在 root 范围内才允许后续操作，否则标记「目标不可达」
- 不递归跟随，防止循环引用导致无限遍历
- Windows 上解析失败时降级（记录警告、不阻塞列表）

### 2.5 SQLite 驱动选型

选用 **`modernc.org/sqlite`**（纯 Go 实现），无需 CGO，`go build` 在任意平台、任意目标交叉编译均零工具链依赖。相比之下 `mattn/go-sqlite3` 需要 CGO + gcc，Windows 上还需 MinGW，不利于交付。

## 3. 整体架构

```
┌───────────────────────────────────────────────┐
│                Browser (Vite SPA)              │
└───────────────────────┬───────────────────────┘
                        │ HTTP/REST
┌───────────────────────▼───────────────────────┐
│                  Go HTTP Server                │
│  中间件管道: Recovery → RequestID → Logger →   │
│  RateLimit → CORS → Auth → PathGuard           │
│                      │                         │
│  ┌───────────────────▼────────────────────┐   │
│  │            Router (chi)                 │   │
│  └──┬────────┬─────────┬──────────┬────────┘   │
│  ┌──▼──┐ ┌───▼───┐ ┌───▼────┐ ┌───▼────┐       │
│  │Auth │ │ File  │ │Bookmark│ │ System │ Handler│
│  └──┬──┘ └───┬───┘ └───┬────┘ └───┬────┘       │
│  ┌──▼────────▼─────────▼──────────▼────────┐   │
│  │              Service Layer              │   │
│  │  (auth / file / bookmark / search)      │   │
│  └──┬──────────────────┬───────────────────┘   │
│  ┌──▼──────┐    ┌───────▼─────────┐            │
│  │  Store  │    │  FS 抽象层        │           │
│  │(SQLite) │    │ (Linux/Windows)  │           │
│  └─────────┘    └──────────────────┘           │
└────────────────────────────────────────────────┘
```

- **Handler 层**：解析/校验请求，调用 service，统一封装响应
- **Service 层**：业务逻辑，不直接拼路径，统一经 `SafeResolve` 取得安全的本地路径
- **Store 层**：SQLite 元数据（用户、会话、收藏夹）
- **FS 抽象层**：封装文件系统操作与平台差异

元数据（用户/会话/收藏夹）走 SQLite；文件内容操作直接走 OS 文件系统，二者职责分离。

## 5. 配置设计

### 5.1 配置项

| 配置项 | 环境变量 | 启动参数 | 默认值 | 说明 |
|--------|----------|----------|--------|------|
| 监听地址 | `FLIST_ADDR` | `--addr` | `:8080` | HTTP 监听地址 |
| 根目录 | `FLIST_ROOT` | `--root` | **必填，无默认** | 允许管理的根目录；缺失则启动失败 |
| 数据目录 | `FLIST_DATA` | `--data` | `./data` | SQLite 及日志存放目录 |
| 管理员用户名 | `FLIST_ADMIN_USER` | `--admin-user` | `admin` | 初始管理员用户名 |
| 管理员密码 | `FLIST_ADMIN_PASS` | `--admin-pass` | 首次启动随机生成并打印日志 | 初始管理员密码 |
| 会话有效期 | `FLIST_SESSION_TTL` | `--session-ttl` | `24h` | 会话过期时间 |
| 长路径支持 | `FLIST_LONG_PATH` | `--long-path` | `false` | 启用 Windows 长路径（`\\?\`），仅 Windows 有效 |
| 上传大小上限 | `FLIST_MAX_UPLOAD` | `--max-upload` | `0`（不限） | 单文件上传上限，0 表示不限 |

> **安全要点**：`--root` 不再提供默认值。默认 `/` 会把整个文件系统暴露出去，风险过高；强制显式指定可避免误操作。

### 5.2 优先级

启动参数 > 环境变量 > 默认值

## 6. 数据库设计 (SQLite)

仅存储元数据。开启 **WAL 模式** 并设置合理的 `busy_timeout`，写连接限制为单连接以规避 `database is locked`（详见 9.5 并发控制）。

### 6.1 用户表 `users`

```sql
CREATE TABLE users (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    username    TEXT NOT NULL UNIQUE,
    password    TEXT NOT NULL,          -- bcrypt hash
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 6.2 会话表 `sessions`

采用**不透明会话令牌**（见认证设计），令牌仅是随机 ID，所有状态存于此表：

```sql
CREATE TABLE sessions (
    id          TEXT PRIMARY KEY,       -- 随机生成的 session token（哈希后存储）
    user_id     INTEGER NOT NULL,
    expires_at  DATETIME NOT NULL,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_seen   DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE INDEX idx_sessions_user ON sessions(user_id);
CREATE INDEX idx_sessions_expires ON sessions(expires_at);
```

过期会话由后台定时任务周期清理。

### 6.3 收藏夹表 `bookmarks`

```sql
CREATE TABLE bookmarks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER NOT NULL,
    name        TEXT NOT NULL,          -- 显示名称
    path        TEXT NOT NULL,          -- 相对 root 的 API 路径（/ 分隔）
    sort_order  INTEGER DEFAULT 0,      -- 排序权重
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE INDEX idx_bookmarks_user ON bookmarks(user_id);
```

> **设计变更**：收藏夹存「相对 root 的路径」而非绝对路径。这样 root 迁移后收藏仍有效，且天然不会指向 root 之外。

> **注意**：SQLite 的 `DEFAULT CURRENT_TIMESTAMP` 仅在 INSERT 生效，UPDATE 不会刷新。`updated_at` 由应用层在更新时显式赋值（或建触发器），不能依赖默认值。

## 7. API 接口设计

统一响应格式：

```json
{ "code": 0, "message": "success", "data": {} }
```

错误时 `code` 非零，`message` 为错误描述。

### 7.1 认证模块

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/auth/login` | 登录，返回会话令牌 | 否 |
| POST | `/api/auth/logout` | 登出，删除当前会话 | 是 |
| PUT | `/api/auth/password` | 修改密码（修改后吊销其他会话） | 是 |
| GET | `/api/auth/me` | 获取当前用户信息 | 是 |

**POST /api/auth/login**

```json
// Request
{ "username": "admin", "password": "xxx" }
// Response
{ "code": 0, "data": { "token": "<opaque>", "expires_at": 1700000000 } }
```

令牌由前端存储（建议 `HttpOnly` Cookie 或前端内存），后续请求经 `Authorization` 头或 Cookie 携带。

| 方法 | 路径 | 说明 | 认证 |

```
Query: ?path=/documents&sort=name&order=asc&page=1&page_size=200
```

```json
// Response
{
  "code": 0,
  "data": {
    "path": "/documents",
    "total": 1532,
    "page": 1,
```

## 8. 逻辑设计

### 8.1 文件操作逻辑

**列表与浏览**
- 前端传相对路径，后端经 `SafeResolve` 拼接 root 得到安全的本地路径
- 返回名称、类型、大小、修改时间、权限模式等元信息，支持按名称/大小/时间排序
- **分页返回**：大目录（数万条目）一次性返回会拖慢响应并占用内存，统一分页
- 隐藏文件默认不显示，可由参数开启

**复制与移动**
- 支持批量；复制目录递归处理
- 移动优先 `os.Rename`（同分区原子操作），跨分区退化为「复制 + 删除」
- 目标存在同名时返回冲突错误，由前端确认是否覆盖

**删除**
- 支持批量；目录递归删除
- 二次确认由前端负责，后端直接执行
- 递归操作设最大深度（默认 100 层）防御异常结构

**搜索**
- 基于文件名的子串/通配匹配，支持递归与非递归
- **串行 `filepath.WalkDir` 遍历**：NAS 多为机械盘，目录遍历是 IO 密集型，并行遍历会加剧磁盘寻道抖动反而更慢
- 设超时（默认 30s）+ 结果数量上限，经 `context` 控制取消，避免大目录树拖垮服务

**上传（分片）**
- 三段式协议：`init`（声明文件名/总大小/分片数，返回 `upload_id`）→ `chunk`（按序号上传分片）→ `complete`（校验并合并）
- 分片**流式写入临时目录**，不将整文件读入内存
- `complete` 时合并到目标并清理临时文件；支持断点续传（重传缺失分片）
- 合并前检查磁盘剩余空间，不足则拒绝

**下载与预览**
- 下载用 `http.ServeContent`，自动支持 **HTTP Range**（断点续传、音视频拖拽）、`ETag`、`Content-Type` 协商
- 预览：文本返回前 N 行/字节，图片返回原图或缩略图

**打包下载**
- `POST /api/fs/archive`，body 传多个文件/目录路径，服务端**流式生成 zip** 边写边发，不在内存或磁盘暂存整个压缩包
- 每个条目经 `SafeResolve` 校验，目录递归加入归档
- 响应头 `Content-Disposition: attachment`；因流式生成无法预知总大小，不返回 `Content-Length`

### 8.2 收藏夹逻辑

- 存「相对 root 的 API 路径」，root 迁移不失效、不会越界
- 浏览时检测目标是否仍存在，不存在标记为失效
- 同一路径不允许重复收藏
- 拖拽排序由前端发起 `reorder` 批量更新

### 8.3 认证逻辑

- 首次启动：库中无用户时，用配置的用户名密码创建管理员；密码未配置则随机生成并打印日志
- 登录：校验用户名密码，成功则生成随机会话令牌，**哈希后**存入 `sessions` 表，明文返回给客户端
- 校验：中间件取令牌哈希后查表，检查存在且未过期，刷新 `last_seen`
- 登出：删除对应会话记录；改密码：吊销该用户其他会话
- 过期会话由后台定时任务清理

> **设计变更（去 JWT）**：原设计用无状态 JWT 却又建表逐请求查 `revoked`，既丢了 JWT 无状态的优势、又背上了复杂度。NAS 单/少用户场景下，**不透明会话令牌**更简单：吊销天然支持（删记录即可），无需密钥管理与签名校验。

## 9. 安全设计

### 9.1 路径穿越防护（核心）

所有文件操作的安全基石：确保解析后的真实路径不逃逸出 root。

```text
SafeResolve(rootReal, userPath):
  1. 清理：在 userPath 前补 "/" 后 filepath.Clean，消解 .. / . 与越根尝试
  2. 拼接：filepath.Join(rootReal, cleaned)
  3. 解析符号链接得真实路径；若目标尚不存在（创建类操作），
     则解析其父目录再拼 basename
  4. 前缀检查：真实路径必须等于 rootReal 或以 rootReal+分隔符 开头，
     否则拒绝（ErrPathTraversal）
```

关键要点：
- **前缀检查必须在符号链接解析之后**，否则可借 symlink 逃逸
- **对尚不存在的目标**（mkdir/touch/upload/copy/move 的目的地）按「父目录解析 + basename」处理，否则会因路径不存在而直接失败
- `rootReal` 在启动时解析一次并缓存，避免每请求重复解析
- 统一在 handler 入口调用；`PathGuard` 中间件作为兜底，拦截含 `..` 或绝对路径的参数

### 9.2 认证安全

| 措施 | 说明 |
|------|------|
| 密码哈希 | bcrypt（cost 默认 10，弱 CPU 设备可下调）；注意 bcrypt 有 72 字节截断 |
| 会话令牌 | 足够长的随机值，哈希后入库；不在库中存明文 |
| 登录限流 | 同 IP 连续失败 5 次锁定 15 分钟（状态持久化，重启不丢） |
| 密码强度 | 改密强制 ≥8 位且含字母与数字 |

### 9.3 请求安全

| 措施 | 说明 |
|------|------|
| 全局限流 | 每 IP 每秒最多 50 请求（令牌桶） |
| 写操作限流 | 上传/复制/删除等写操作每秒最多 10 请求 |
| 请求体限制 | 非上传接口 body ≤ 1MB；上传按配置限制 |
| CORS | 同源嵌入部署默认**无需 CORS**；仅当前后端分离调试时按需开启白名单 |
| Content-Type 校验 | 严格校验与接口预期匹配 |

### 9.4 文件操作安全

| 措施 | 说明 |
|------|------|
| 系统关键路径黑名单 | Linux: `/proc`、`/sys`、`/dev`、`/boot` 等；Windows: `C:\Windows`、`Program Files` 等 |
| 符号链接限制 | 解析后目标必须仍在 root 内 |
| 文件名校验 | 按平台禁危险字符与保留名（见 2.2） |
| 递归深度保护 | 递归删除/复制设最大深度 |
| 并发写保护 | 进程内路径级锁（见 9.5） |
| 磁盘空间检测 | 上传/复制前检查剩余空间 |

### 9.5 并发控制

flist 是**单进程**服务，并发来自 goroutine 而非跨进程。因此用**进程内路径级互斥锁**（按路径维护 `sync.Mutex`），对同一目标的写操作串行化即可，简单高效、无残留锁文件。

> **设计变更**：原设计用 OS 级 `flock`/`LockFileEx` + `.lock` 文件，那是为跨进程场景准备的，对单进程是过度设计，且原实现锁的是 `.lock` 文件、未真正保护目标文件。

SQLite 侧并发要点：
- 开启 **WAL 模式**（读写并发更好）
- 设置 `busy_timeout`（如 5s），减少瞬时锁冲突报错
- 写连接 `SetMaxOpenConns(1)`，避免多写连接竞争导致 `database is locked`

### 9.6 传输安全

- 生产环境建议前置反向代理（Nginx/Caddy）做 TLS 终结
- 统一安全响应头：`X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY`、`Content-Security-Policy`、HTTPS 下 `Strict-Transport-Security`
- CSP 需结合 Vite 产物实测（内联 style/script 可能需调整策略）

### 9.7 日志与审计

- 所有写操作（创建/删除/移动/复制/上传）记录审计日志：时间、操作类型、路径、来源 IP、结果
- 登录失败记录 IP 与尝试的用户名
- 日志输出到文件，按天轮转

## 13. 构建与部署

### 13.1 构建流程

```bash
# 1. 前端构建
cd frontend && npm run build       # 产出 frontend/dist/
# 2. 复制到嵌入目录
cp -r frontend/dist web/dist       # Windows: xcopy /E /I
# 3. 后端构建（embed 自动打包 web/dist/）
go build -o flist ./cmd/flist
```

`web/embed.go` 通过 `//go:embed dist` 嵌入前端产物，后端用 `http.FS` 提供静态资源。

> **SPA 路由 fallback**：前端使用 history 路由时，对未命中静态资源的非 API 路径需回退返回 `index.html`，否则刷新子路由会 404。embed 静态服务需处理该回退逻辑。

### 13.2 跨平台交叉编译

Go 原生支持，通过 `GOOS`/`GOARCH` 控制目标平台。Makefile 提供 `build-linux`、`build-windows`、`build-all` 等目标，前端构建作为前置依赖。因选用纯 Go SQLite 驱动，交叉编译无需额外工具链。

### 13.3 启动示例

```bash
# Linux
./flist --root /mnt/nas-storage --admin-pass "s3cureP@ss"
# Windows
flist.exe --root "D:\share" --admin-pass "s3cureP@ss"
```

`--root` 为必填项，未提供时启动失败。

### 13.4 Linux 部署

- 以非 root 用户运行，靠文件系统权限约束访问范围
- 前置 Nginx/Caddy 做 TLS
- systemd 管理生命周期，配合 `ProtectSystem=strict` + `ReadWritePaths` 收窄可写范围
- 定期备份数据目录（主要是 SQLite 文件，WAL 模式下注意一并备份 `-wal`/`-shm` 或先 checkpoint）

### 13.5 Windows 部署

- 推荐注册为 Windows 服务实现开机自启与后台运行，可用内置 `sc.exe` 或更灵活的 NSSM
- 通过 `golang.org/x/sys/windows/svc` 判断是否以服务方式运行，分别走服务/控制台入口
- 防火墙放行监听端口

### 13.6 数据目录结构

```
<DATA_DIR>/
├── data.db              # SQLite 主库
├── data.db-wal          # WAL（运行时）
├── uploads.tmp/         # 分片上传临时目录
└── logs/                # 日志（可选）
```

Linux 默认 `./data`，Windows 默认 `.\data`，可由 `--data` / `FLIST_DATA` 覆盖。

## 14. 跨平台测试策略

### 14.1 测试矩阵

| 维度 | Linux | Windows |
|------|-------|---------|
| 路径转换 | `/mnt/data` | `D:\share` |
| 文件名校验 | `/`、`\0` | + `<>:"/\|?*`、保留名 |
| 隐藏文件 | `.` 前缀 | `FILE_ATTRIBUTE_HIDDEN` |
| 符号链接 | 解析 + 前缀检查 | 降级处理 |
| 磁盘空间 | `Statfs` | `GetDiskFreeSpaceEx` |
| 长路径 | 无限制 | `\\?\` 前缀 |
| 后台运行 | systemd | Windows 服务 |

### 14.2 测试重点

- **路径安全**：构造各类穿越 payload（`..`、绝对路径、symlink 逃逸、编码绕过），验证 `SafeResolve` 全部拦截
- **创建类操作**：验证目标不存在时 mkdir/touch/upload/copy/move 能正常解析并落地
- **并发**：多 goroutine 同时写同一目标，验证路径级锁正确串行化
- **分片上传**：乱序分片、缺片重传、合并校验
- **平台隔离代码**：用 build tags + `t.TempDir()` 编写平台特定单测
- **CI 矩阵**：GitHub Actions 在 `ubuntu-latest` 与 `windows-latest` 同时运行 `go test ./...`
