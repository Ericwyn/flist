# Phase 2 详细设计：基础文件操作

> 对齐 `0.backend-design.md` 与 `1.开发阶段划分.md`，承接 Phase 0 的服务骨架 / `SafeResolve` 与 Phase 1 的只读路径。本文聚焦 Phase 2「写」与「搜索」的可落地细节。复制（copy）与收藏夹属 Phase 3，本阶段仅把**递归复制原语**下沉到 `util` 供 Phase 3 复用，不开放 `/api/fs/copy` 接口。

## 1. 目标与边界

**目标**：实现文件 / 目录的创建、重命名、移动、删除，以及基于文件名的搜索。前端可经右键菜单与确认弹窗完成这些操作。

**本阶段包含**：
- `POST /api/fs/mkdir`：创建目录
- `POST /api/fs/touch`：创建空文件
- `POST /api/fs/move`：移动 / 重命名（批量，同分区 `rename`，跨分区降级复制 + 删除）
- `DELETE /api/fs/delete`：删除（批量、递归、最大深度保护）
- `GET /api/fs/search`：搜索（串行遍历 + 超时 + 结果数量上限）
- 文件名校验（跨平台非法字符 + Windows 保留名，build tag 隔离）
- 写操作的**结构化审计日志**（slog，横切项起步）
- 写操作限流接入（`10/s per IP`，复用 Phase 0 已建的 `NewWriteRateLimiter`）
- 前端：右键菜单（新建 / 重命名 / 删除）、名称输入弹窗、删除确认弹窗、搜索框与结果视图

**本阶段不包含**：复制接口与收藏夹（Phase 3）、上传（Phase 4）、打包下载与多选批量交互（Phase 5）、系统信息（Phase 6）。Phase 2 的批量入参（move/delete 的数组）是协议层面的批量，**前端多选交互**留到 Phase 5。

> **关于移动（move）的前后端边界**：后端 `POST /api/fs/move` 本阶段**完整实现**跨目录批量移动（dst 是已存在目录则移入、不存在则重命名）。但**前端只暴露「重命名」**（同目录 move 的封装），**不提供「移动到其他目录」的 UI 入口**。原因：跨目录移动的自然交互是「剪切 + 粘贴」，与 Phase 3 的复制粘贴共用同一套剪贴板状态；为避免在 Phase 2 做一版「移动到…」目录选择器、到 Phase 3 又被剪贴板交互推倒重来，故把 cut（=move）/ paste 形态的移动与 copy 一起放到 Phase 3 实现。本阶段前端 move 能力仅以 `rename` 形式落地。

**验收标准**：可在前端完成新建、重命名、删除；搜索正常返回且大目录不拖垮服务；非法文件名被拦截。

## 2. 关键设计取舍（已确认）

| 取舍点 | 决策 | 理由 |
|--------|------|------|
| move 跨分区 | **现在实现** copy+delete 回退；递归复制原语下沉到 `util.CopyPath`，Phase 3 的 copy 接口复用 | root 跨多个挂载点（bind mount 等）时 `rename` 会返回 `EXDEV`，必须回退否则 move 直接失败 |
| 审计日志 | **结构化 slog 事件**（user / action / path / result / request_id / ip） | 随 Phase 0 已建的 slog 体系自然落地，零额外依赖；落库留到有查询需求时 |
| 批量语义 | **尽力而为**，逐项返回 `{src, ok, error}` | 贴近主流文件管理器；单条失败不影响其余项 |
| 同名冲突 | 返回 `2004 file_exists`，**不覆盖** | Phase 2 主要场景是重命名；跨目录覆盖留到 Phase 3 剪贴板交互 |

## 4. 数据模型（model/file.go 追加）

```go
// OpResult 批量写操作的单条结果。
type OpResult struct {
    Src   string `json:"src"`             // 操作对象的 API 路径
    OK    bool   `json:"ok"`
    Error string `json:"error,omitempty"` // 失败时的错误码名（如 "file_exists"）
}

// SearchHit 单条搜索命中。
type SearchHit struct {
    Path    string    `json:"path"`     // 相对 root 的 API 路径（含文件名）
    Name    string    `json:"name"`
    Type    string    `json:"type"`     // file | dir
    Size    int64     `json:"size"`
    Mode    string    `json:"mode"`
    ModTime time.Time `json:"mod_time"`
}

// SearchResult 搜索接口返回体。
type SearchResult struct {
    Query     string      `json:"query"`
    Base      string      `json:"base"`       // 搜索起点 API 路径
    Items     []SearchHit `json:"items"`
    Truncated bool        `json:"truncated"`  // 命中达到上限被截断
    TimedOut  bool        `json:"timed_out"`  // 遍历超时提前结束
}
```

## 5. 接口设计

统一响应信封同前序阶段。所有接口需认证，经 `Auth → PathGuard` 管道；写接口额外经写限流。写接口的 body ≤ 1MB（`decodeJSON` 已限制）。

### 5.1 POST /api/fs/mkdir

```json
// Request  { "path": "/documents/newdir" }
// Response { "code":0, "data": { "path":"/documents/newdir" } }
```
- 父目录必须存在；basename 经文件名校验
- 目标已存在 → `2004 file_exists`
- 仅创建**单层**目录（`os.Mkdir`，非 `MkdirAll`）：父目录不存在 → `2001 path_not_found`，语义清晰、避免误建深层路径

### 5.2 POST /api/fs/touch

```json
// Request  { "path": "/documents/empty.txt" }
// Response { "code":0, "data": { "path":"/documents/empty.txt" } }
```
- 父目录必须存在；basename 经文件名校验
- 目标已存在 → `2004 file_exists`（不 truncate 已有文件）
- 用 `os.OpenFile(..., O_CREATE|O_EXCL, 0644)` 保证「不存在才创建」的原子性

### 5.3 POST /api/fs/move

```json
// Request  { "src": ["/a/old.txt"], "dst": "/a/new.txt" }
// Response { "code":0, "data": { "results": [ { "src":"/a/old.txt", "ok":true } ] } }
```

**dst 语义**（与 `mv` 一致）：
- `dst` 解析后**是已存在目录** → 把每个 `src` 移动进该目录，落点为 `dst/basename(src)`（移动 / 整理）
- `dst` **不存在** → 仅当 `len(src)==1` 时合法，视为「重命名 / 移动到指定名」；`len(src)>1` 时每项标记失败（`not_a_dir`）
- `dst` 存在但**是文件** → 每项失败（`not_a_dir`，不允许覆盖文件）

**单项规则**：
- 落点 basename 经文件名校验（重命名场景下校验新名）
- 落点已存在 → 该项 `2004 file_exists`
- src 不存在 → 该项 `2001`
- src 即 root 或落点逃逸 → 该项 `2002`
- 优先 `os.Rename`；遇 `EXDEV`（跨分区）→ `CopyPath` + `RemovePath` 回退
- 不允许把目录移动进其自身子树（`dst` 在 `src` 之内）→ 该项 `4000 bad_request`

整体始终返回 `code:0` 与逐项 `results`（尽力而为）；仅当请求体非法（缺字段、空 src）时返回 `4000`。

### 5.4 DELETE /api/fs/delete

```json
// Request  { "paths": ["/a/x.txt", "/a/sub"] }
// Response { "code":0, "data": { "results": [ {"src":"/a/x.txt","ok":true}, {"src":"/a/sub","ok":true} ] } }
```
- 逐项 `SafeResolve` → 递归删除（`RemovePath`，最大深度 100）
- **不允许删除 root 自身** → 该项 `4000 bad_request`
- 目标不存在 → 该项 `2001`
- 二次确认由前端负责，后端直接执行

### 5.5 GET /api/fs/search

```
Query:
  path         搜索起点，默认 "/"
  q            关键字（必填，去空格后非空）
  recursive    true | false，默认 true
  show_hidden  true | false，默认 false
  limit        结果上限，默认 500，最大 1000
```
- 起点 `SafeResolve` → 必须是目录，否则 `2008 not_a_dir`
- 匹配规则：`q` 含 `*` 或 `?` → 按 `path.Match` 通配匹配文件名；否则按**大小写不敏感子串**匹配
- 串行 `filepath.WalkDir`（NAS 多机械盘，并行遍历加剧寻道抖动）
- 超时默认 **30s**（`context.WithTimeout`），超时 → `timed_out=true` 并返回已命中结果
- 命中数达 `limit` → `truncated=true` 并停止遍历
- 隐藏文件按 `show_hidden` 过滤；符号链接**不跟随**（避免循环），仅按名字匹配自身
- 单条 `Lstat` 失败跳过，不中断整体

```json
{ "code":0, "data": {
  "query":"报告", "base":"/documents",
  "items":[ {"path":"/documents/年度报告.pdf","name":"年度报告.pdf","type":"file","size":2048,"mode":"0644","mod_time":"..."} ],
  "truncated":false, "timed_out":false
}}
```

## 6. 文件名校验（util/name.go + 平台分支）

```go
// ValidateName 校验单个文件 / 目录名（非路径）。先公共规则，再平台规则。
func ValidateName(name string) error
```

**公共规则**（所有平台）：
- 非空、长度 ≤ 255 字节
- 不含 `/`（路径分隔符）与 NUL（`\x00`）
- 不为 `.` 或 `..`

**Linux**（`name_linux.go`）：无额外限制（公共规则已足够）。

**Windows**（`name_windows.go`）：
- 额外禁字符：`< > : " \ | ? *`
- 保留名（不分大小写，含带扩展名形式如 `CON.txt`）：`CON PRN AUX NUL COM1-9 LPT1-9`
- 不可以空格或点 `.` 结尾

> 跨平台一致性说明：Phase 2 按**运行平台**校验（与 `0.backend-design.md` 2.2「校验函数按平台分支」一致）。NAS 若需被 Windows 客户端访问，可在部署侧约束；强制全平台按 Windows 规则校验留作可选增强，不在本阶段强加。

校验失败统一映射 `2006 name_invalid`。

## 7. 文件操作原语（util/fsop.go + 平台分支）

```go
const maxRecursionDepth = 100 // 递归深度保护，防御异常 / 恶意目录结构

// CopyPath 递归复制 src 到 dst（dst 须不存在）。文件复制内容与权限位；
// 目录复制其下全部条目。符号链接按 0.backend-design.md 不创建，遇到时复制其指向的内容快照（Lstat 判定后跳过或按需处理）。
func CopyPath(src, dst string) error

// RemovePath 递归删除 path（深度受限）。
func RemovePath(path string) error

// MovePath 移动 src 到 dst：优先 os.Rename；遇跨分区（EXDEV）回退 CopyPath + RemovePath。
func MovePath(src, dst string) error

// isCrossDevice 由平台文件实现：Linux 判 syscall.EXDEV，Windows 判 ERROR_NOT_SAME_DEVICE。
func isCrossDevice(err error) bool
```

要点：
- `CopyPath` 对目录递归时累加深度，超过 `maxRecursionDepth` 返回错误
- 复制保留权限位（`fi.Mode().Perm()`），目录用 `0755` 兜底
- 复制过程中遇符号链接：本阶段不重建链接，记录并跳过（与「flist 自身不创建符号链接」一致）
- `MovePath` 回退路径：复制成功后再删源；复制中途失败则清理已复制的 dst，避免半成品

## 9. 审计日志

写操作成功 / 失败均记一条结构化日志（不阻塞主流程）：

```go
logger.Info("audit",
  slog.String("action", "mkdir"|"touch"|"move"|"delete"),
  slog.String("path", apiPath),       // move 记 src→dst
  slog.String("user", username),
  slog.String("result", "ok"|"fail"),
  slog.String("request_id", reqID),
  slog.String("ip", clientIP),
)
```

- handler 持有 `*slog.Logger`，从 ctx 取 user / request_id，从请求取 ip
- 批量操作按**每项**记一条，便于审计追溯
- search 属读操作，不记审计（已被访问日志覆盖）

## 12. 前端改造

**数据层**
- `lib/api.ts` 追加：
  - `fs.mkdir(path)`、`fs.touch(path)`
  - `fs.move(src: string[], dst)`、`fs.rename(fromPath, newName)`（move 的便捷封装：`dst = parent(fromPath)/newName`）
  - `fs.delete(paths: string[])`
  - `fs.search(base, q, {recursive, showHidden, limit})`
- `types.ts` 追加 `OpResult` / `SearchHit` / `SearchResult`
- `fsStore.ts` 追加：`mkdir/touch/rename/remove` action（成功后 `refresh`）；搜索态 `searchQuery / searchResults / searching / searchOpen`，`runSearch` / `clearSearch`

> `fs.move` 已在 api 层就绪并支持跨目录批量移动，但本阶段 `fsStore` **只暴露 `rename`**（同目录 move 封装），不暴露通用 `move` action，右键菜单也无「移动到…」项（见 §1 边界说明）。跨目录移动的 cut/paste 入口随 Phase 3 剪贴板交互一并落地，届时直接复用现成的 `fs.move`。

**UI**
- `InputModal`：通用名称输入（新建文件夹 / 新建文件 / 重命名），前端做基础非空与非法字符即时提示，最终以后端校验为准（`2006` → 弹出错误）
- `ConfirmModal`：删除二次确认，列出目标名与「不可恢复」提示
- `SearchBar`：工具栏搜索入口，输入关键字回车触发，结果以列表覆盖当前目录视图；点击结果项：目录则 `navigate`，文件则定位到所在目录或直接预览；可清除回到普通浏览
- `FileBrowser` 右键菜单：
  - 空白处：新建文件夹、新建文件、刷新
  - 文件 / 目录：打开 / 预览、重命名、删除（danger）、下载（文件）、属性
- 操作完成后刷新当前目录；`2004` 提示「已存在」、`2006` 提示「名称非法」、`2003` 提示「权限不足」

## 13. 测试点

**后端**
- `util.ValidateName`（linux）：空 / 含 `/` / 含 NUL / `.` / `..` / 超长 → 拒绝；正常名通过
- `util.CopyPath`：复制文件（内容 + 权限）、复制目录树、dst 已存在报错、超深度报错
- `util.RemovePath`：删文件、删目录树、超深度报错
- `util.MovePath`：同目录 rename、跨目录移动；（跨分区回退在 CI 难造，靠 copy+remove 单测覆盖回退路径的两段）
- `service.Mkdir/Touch`：成功、父目录不存在、已存在冲突、非法名
- `service.Move`：重命名、移动进目录、落点冲突、src 不存在、移动目录进自身子树被拒、批量逐项结果
- `service.Delete`：删文件 / 目录、root 自身被拒、不存在项标记失败、批量逐项结果
- `service.Search`：子串命中、通配命中、recursive=false 仅当层、limit 截断 `truncated`、隐藏过滤、起点非目录报错、context 取消即时返回
- `handler` e2e（httptest）：mkdir→list 出现、touch→stat 命中、rename→旧无新有、delete→消失、search 返回、非法名 2006、冲突 2004、写接口需认证、PathGuard 拦越界

**端到端冒烟**：登录 → 新建目录 → 新建文件 → 重命名 → 搜索命中 → 删除 → 列表刷新一致。

## 14. 安全要点

- 所有写接口入口 `SafeResolve`，前缀检查在符号链接解析之后（复用 Phase 0）
- 创建类目标「不存在」由 `SafeResolve` 的父目录解析分支处理
- 文件名校验拦截非法字符 / 保留名，防止越权写入与平台异常
- 删除 / 移动**禁止作用于 root 自身**；移动禁止进入自身子树（防自包含循环）
- 递归操作最大深度 100，防御异常 / 恶意目录结构
- 写限流 10/s per IP，缓解写放大攻击
- 审计日志留痕，便于事后追溯
