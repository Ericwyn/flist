# Phase 5 详细设计：批量操作与打包下载

> 对齐 `0.backend-design.md`、`1.开发阶段划分.md` 与 `5.存储驱动抽象设计.md`。承接 Phase 0 服务骨架、Phase 1 只读 / 下载、Phase 2 写 / 搜索、Phase 3 复制 / 收藏夹、Phase 4 分片上传。本文聚焦 Phase 5「批量操作与打包下载」。

## 1. 目标与边界

**目标**：在前端打通「多选 → 批量操作」的完整交互，并新增服务端流式 zip 打包下载。

**本阶段包含**：
- **打包下载**：`POST /api/fs/archive`，选中多文件 / 文件夹一键下载为 zip；服务端**流式生成**（边写边发），不在内存或磁盘暂存整个压缩包。
- **批量删除 / 批量移动 / 批量复制**：复用 Phase 2/3 既有数组入参接口（`move`/`delete`/`copy`），本阶段补齐前端多选入口去驱动它们。
- **批量上传**：复用 Phase 4 的 `uploadStore` 多文件入队能力，无需新增后端。
- **前端多选交互**：单击单选、Ctrl/Cmd 切换、Shift 范围选择、Ctrl/Cmd+A 全选、Esc 清空；批量右键菜单与工具栏。

**本阶段不包含**：系统信息面板（Phase 6）；打包下载的断点续传 / 后台任务化（zip 流式生成无法预知 Content-Length，不支持 Range）。

**验收标准**：可批量删除 / 批量下载；zip 流式生成、内容完整、不占用大量内存；多选交互符合主流文件管理器约定。

## 2. 关键设计取舍（已与用户确认）

| 取舍点 | 决策 | 说明 |
|--------|------|------|
| 打包下载浏览器触发 | **fetch POST → Blob → 触发下载（方案 A）** | 兼容所有浏览器、错误处理干净（可先读到 JSON 错误再决定是否下载）、实现最简。代价：整个 zip 先缓存在浏览器内存，超大包（数 GB）占内存。NAS 单 / 少用户场景可接受 |
| zip 压缩策略 | **智能压缩** | 已压缩的媒体 / 归档类（mp4/jpg/png/zip/gz/docx 等）用 `Store` 不二次压缩，其余（文本 / 文档等）用 `Deflate`。兼顾 NAS 弱 CPU 与压缩率 |
| 打包是否含隐藏文件 | **包含（showHidden=true）** | 「下载这个文件夹」语义下应是忠实副本，含 `.gitignore` 等隐藏项；与列表视图的隐藏开关解耦 |
| 符号链接 / 特殊文件 | **跳过（仅打包普通文件与目录）** | 不跟随、不打包符号链接 / 设备文件，避免循环引用与越界；打包时静默跳过 |
| 流式错误处理 | **写首字节前预检，写中途出错只能中断连接** | 预检（Stat 所有顶层路径）通过后才发送响应头与 zip 字节；预检失败返回标准 JSON 错误码。中途出错（文件被并发删除 / 读失败）记日志、不补齐 zip 中央目录，客户端据此判定损坏 |

> **为何方案 A 而非原生表单 / File System Access API**：表单提交错误反馈弱（出错只能静默或粗糙提示）；File System Access API 仅 Chromium 系支持，需降级分支。方案 A 在「兼容性 + 错误可控 + 实现复杂度」上最优，超大包占内存是可接受的折中。

## 3. 现状盘点（已就绪 / 待补）

| 能力 | 状态 |
|------|------|
| 后端 `move` / `delete` / `copy` 批量数组入参 | **已实现**（Phase 2/3，逐项「尽力而为」返回 `OpResult[]`） |
| 后端多文件分片上传 | **已实现**（Phase 4，前端 `uploadStore` 逐文件复用单文件链路） |
| `backend.Open`（读单文件）/ `backend.Walk`（递归遍历） | **已实现**（Phase 1 / Phase 2 搜索复用） |
| **`POST /api/fs/archive` 流式 zip** | **全空白**（本阶段核心后端工作） |

| **前端多选模型**（当前单选 `selected: string \| null`） | **需重构为多选集合 + 锚点** |

> **archive 不需要新的 `storage.Backend` 接口**：它只组合既有的 `Open`（读文件字节）+ `Walk`（递归列举），纯在 `FileService` 层做流式 zip 编排，完全符合「service 编排、backend 物理存取」的抽象原则。

## 4. 职责划分

延续存储抽象的「service 编排、backend 物理存取」原则：

| 留在 service（后端无关、policy） | 复用 backend（后端相关、物理） |
|---|---|
| 顶层路径预检与解析、zip 条目命名 / 去重 | `Stat`：路径校验与类型判定 |
| 目录递归编排、流式 zip 写入、智能压缩选择 | `Walk`：递归列举（local 用 `WalkDir`） |
| 跳过符号链接 / 特殊文件、空目录保留 | `Open`：打开普通文件供读取字节 |

## 5. 打包下载接口

### 5.1 `POST /api/fs/archive`

```json
// Request
{ "paths": ["/documents/a.txt", "/photos/vacation"], "name": "download" }
```

```
// Response（成功）
HTTP/1.1 200 OK
Content-Type: application/zip
Content-Disposition: attachment; filename*=UTF-8''download.zip
X-Content-Type-Options: nosniff
<流式 zip 字节>
```

- `paths`：相对 root 的 API 路径数组（文件或目录），非空。
- `name`：可选，zip 下载文件名（不含扩展名）；前端缺省时按选择回落（单项取其名，多项取 `flist-download`）。
- **无 `Content-Length`**：流式生成无法预知总大小，故不支持 Range / 断点续传。

### 5.2 处理流程

1. **预检（写首字节前）**：对每个 `paths[i]` 经 `backend.Stat` 校验存在性与越界；任一失败 → 返回标准 JSON 错误（`2001 path_not_found` / `2002 path_traversal` 等），**不发送任何 zip 字节**。其中 root（`/`，含会被归一化为 `/` 的 `""`/`//`/`..` 等形式）**禁止打包**：无明确顶层名、易与其他选中项重名，且「下载整个根」语义不当，直接返回 `4000 bad_request`。
2. **顶层条目命名**：每个顶层路径的 zip 内名取其 `basename`；多个同名（极少，仅当跨目录选择）做 ` (2)` 递增去重。
3. **设响应头** → 创建 `zip.Writer(w)`（`w` 为 `http.ResponseWriter`，flush 即下发）。
4. **逐顶层条目写入**：
   - 文件：以 `basename` 为名写入单条目（按扩展名选 `Store` / `Deflate`）。
   - 目录：写目录条目（名以 `/` 结尾，保留空目录），再 `backend.Walk`（`showHidden=true`）递归：子文件 `Open` 后流式拷入、子目录写目录条目、**符号链接 / 特殊文件跳过**。
5. `zip.Writer.Close()` 写中央目录收尾。

### 5.3 智能压缩

```text
methodFor(name):
  ext := lower(path.Ext(name))
  if util.IsCompressedExt(ext): return zip.Store    // 已压缩，不二次压缩
  else:                         return zip.Deflate   // 文本/文档等，压缩有收益
```

`util.IsCompressedExt` 覆盖：归档（zip/gz/tgz/bz2/xz/7z/rar）、图片（jpg/png/gif/webp/avif）、音视频（mp4/mkv/mov/webm/mp3/aac/flac/...）、zip 容器型文档（docx/xlsx/pptx/jar/apk）、pdf 等。

## 6. 错误码

**无新增错误码**。复用：`2001 path_not_found`、`2002 path_traversal`、`4000 bad_request`（paths 为空，或包含被禁止的 root `/`）。中途流式错误无法用状态码表达，记日志（`--debug` 下输出细节）。

## 7. 路由（server/routes.go）

archive 为只读型重操作（不修改文件），**不套写限流**，仅受全局限流（50/s）：

```go
protected.Post("/fs/archive", fileHandler.Archive)
```

请求体为小 JSON（路径数组），走既有 1MB `decodeJSON` 通道。

## 9. 前端多选模型

### 9.1 状态（fsStore）

`selected: string | null` → `selected: Set<string>`（条目名集合）+ `selectionAnchor: string | null`（Shift 范围锚点）。导航 / 刷新 / 进入搜索时清空选择。

| 方法 | 语义 |
|------|------|
| `selectOne(name)` | 清空其余，仅选 name；置锚点（普通单击） |
| `toggleSelect(name)` | 切换 name 的选中态；置锚点（Ctrl/Cmd 单击） |
| `rangeSelect(name)` | 从锚点到 name（按当前 `entries` 顺序）的连续区间作为选择（Shift 单击） |
| `selectAll()` | 选中当前目录全部条目（Ctrl/Cmd+A） |
| `clearSelection()` | 清空（Esc / 点击空白） |

派生：`selectedEntries = entries.filter(e => selected.has(e.name))`。

### 9.2 交互约定（贴近主流文件管理器）

- **单击**：单选该项。
- **Ctrl/Cmd+单击**：切换该项选中态（多选累加 / 取消）。
- **Shift+单击**：以上次锚点为起点，选中到当前项的连续区间。
- **Ctrl/Cmd+A**：全选（输入框 / 搜索态不拦截）。
- **Esc**：清空选择。
- **右键**：若点击项已在多选集合内（且选中数 > 1）→ 保留选择、弹批量菜单；否则单选该项再弹单项菜单。
- **双击**：打开（目录进入 / 文件预览），不改变多选语义。

### 9.3 批量操作入口

- **工具栏「下载」**：选中单个文件 → 直链下载（原行为）；选中含目录或多项 → 打包下载（archive）。
- **右键批量菜单**（选中数 > 1）：打包下载 / 复制 / 剪切 / 删除。
- **快捷键**：`Ctrl/Cmd+C` 复制选择集、`Ctrl/Cmd+X` 剪切选择集、`Ctrl/Cmd+V` 粘贴到当前目录（复用 Phase 3）、`Delete` 批量删除（确认弹窗显示选中项数）。
- **批量删除确认**：`ConfirmModal` 文案显示「确定删除选中的 N 项吗？」。

### 9.4 打包下载（api.ts）

```ts
fs.archive(paths: string[], name?: string): Promise<void>
//  POST /api/fs/archive（JSON body）→ 非 200 先解析 JSON 错误码抛 ApiError；
//  200 则 resp.blob() → URL.createObjectURL → 触发 <a download> → revoke。
//  下载文件名取 name（回落：单项取其名、多项 flist-download）+ ".zip"。
```

## 10. 并发与安全

- **路径安全**：每个顶层路径与递归子项均经 backend 的 `SafeResolve`（`Stat`/`Walk`/`Open` 内部），不逃逸 root；符号链接越界项在 Walk 中按 Lstat 信息呈现、打包时跳过。
- **资源**：zip 流式生成、单文件流式拷贝，服务端内存占用与单文件下载同量级（不随包大小线性增长）。
- **取消**：`http.ResponseWriter` 对应连接断开时 `r.Context()` 取消，`Walk` 与拷贝循环据此提前退出，不空转。
- **限流**：archive 仅受全局 50/s；批量 move/delete/copy 仍走写限流 10/s（逐请求，非逐项）。
- **审计**：批量写（move/delete）逐项审计（Phase 2 既有）；archive 为只读，记 `--debug` 级别日志，不计入写审计。

## 11. 测试点

**后端**
- `util.IsCompressedExt`：已压缩扩展名命中 Store、其余 Deflate。
- `service.WriteArchive`：
  - 单文件打包 → zip 含 1 条目、内容字节一致。
  - 目录打包 → 含目录条目与全部子文件、相对路径前缀正确（`base/sub/x.txt`）、空目录保留。
  - 多路径混合（文件 + 目录）→ 各顶层名正确、跨目录同名去重。
  - 含隐藏文件 → 打包包含（showHidden=true）。
  - 符号链接 → 跳过、不报错。
  - 智能压缩 → `.txt` 用 Deflate、`.jpg`/`.zip` 用 Store（校验条目 Method）。
- `service.ResolveArchiveTargets`：路径不存在 → `ErrNotFound`；越界 → `ErrTraversal`；root（`/` 及归一化为 `/` 的形式，含与其他路径混选）→ `ErrBadOp`。

**handler e2e**
- 未登录 → 401。
- 预检失败（含不存在路径）→ JSON `2001`，无 zip 字节。
- 成功 → `Content-Type: application/zip`、`Content-Disposition: attachment`、body 为可解压的合法 zip，解压后文件树与内容一致。
- 空 `paths` → `4000`。

**前端 / 端到端冒烟**：多选（单击 / Ctrl / Shift / 全选）→ 批量删除（确认）→ 批量复制粘贴 → 选中多项 / 文件夹打包下载 → 解压校验内容完整。

## 12. 与既有设计的一致性

- archive 复用既有 `Open` / `Walk`，**不新增 backend 接口**，service 编排、backend 物理存取的分层不变。
- 错误词表延续 `storage` 统一错误 + handler `errors.Is` 映射，无新增错误码。
- 批量 move/delete/copy 直接复用 Phase 2/3 的数组入参与 `OpResult[]` 尽力而为语义。
- 统一响应信封、限流、路径安全、SPA 静态托管均复用既有管道，无新增中间件。
