# Spec 写作方法

## 1. Spec 是什么

Spec 是面向工程协作的技术方案文档，用来在动手实现之前，把需求背景、当前现状、设计取舍、实现方案、兼容策略和验证方式讲清楚。

它不是纯 PRD，也不是代码实现记录。

一个好的 spec 应该回答这些问题：

1. 为什么要做这件事？
2. 当前系统是怎么工作的？
3. 这次要解决什么问题？
4. 明确不解决什么问题？
5. 方案有哪些关键取舍？
6. 需要改哪些模块、接口、数据结构或配置？
7. 失败、异常、兼容和回滚怎么处理？
8. 如何验证这个方案是正确的？

一句话概括：

```text
Spec = 背景 + 现状 + 目标 + 非目标 + 设计方案 + 实施计划 + 验证方式
```

---

## 2. 推荐存放位置

推荐统一放在：

```text
docs/spec/
```

常见目录结构：

```text
docs/
  spec/
    260429-semantic-search/
      260429-semantic-search-overview.md
      260429-semantic-search-implementation.md
    260430-cache-optimization/
      260430-cache-optimization-overview.md
    spec-write-doc.md
```

说明：

1. `docs/spec/` 用于存放所有设计方案类文档。
2. 每个较独立的需求或技术改造，建议使用一个单独目录。
3. 通用规范、模板、写作指南可以直接放在 `docs/spec/` 根目录。
4. 已完成的 spec 不建议随意删除，它们是后续理解历史决策的重要依据。

## 3. 命名规范

### 3.1 推荐目录命名

推荐格式：

```text
YYMMDD-feature-name
```

示例：

```text
260429-semantic-search
260430-cache-optimization
260501-session-redis-lock
```

说明：

1. `YYMMDD` 用来表达创建日期，方便按时间排序。
2. `feature-name` 用短横线连接，尽量简短、可读。
3. 目录名可以使用中英文混合，但更推荐英文或项目团队统一约定的命名方式。
4. 如果是中文业务名，也应保持短而明确，例如：

```text
260429-商品融合优化
260430-搜索筛选优化
```

### 3.2 推荐文件命名

具体 spec 文档建议使用完整命名，格式为：

```text
YYMMDD-feature-name-doc-type.md
```

其中：

- `YYMMDD-feature-name` 应与所在目录名保持一致。
- `doc-type` 用来表达文档类型，例如 `overview`、`implementation`、`phase-1`、`metrics`。

较完整的需求建议拆成：

```text
260429-semantic-search-overview.md
260429-semantic-search-implementation.md
```

不推荐在需求目录下使用过于通用的文件名，例如：

```text
overview.md
implementation.md
```

原因是：

1. 单独查看文件名时无法判断它属于哪个需求。
2. 搜索、引用、IDE tab 展示时容易出现多个同名 `overview.md`。
3. 完整文件名更方便在聊天、PR、评审评论和历史记录中直接引用。

含义：

| 文件 | 适用内容 |
| --- | --- |
| `260429-semantic-search-overview.md` | 背景、目标、非目标、当前现状、核心设计、总体方案 |
| `260429-semantic-search-implementation.md` | 具体代码改造点、结构体、接口、迁移步骤、测试细节 |

小需求或单点优化可以只写一个完整命名文件：

```text
260430-cache-optimization-overview.md
```

推荐判断方式：

| 场景 | 推荐形式 |
| --- | --- |
| 只涉及一个小改动，文档不超过 100 行 | `YYMMDD-feature-name-overview.md` 单文件 |
| 涉及多个模块，但实现不复杂 | `YYMMDD-feature-name-overview.md` |
| 涉及多个模块、接口、数据结构或上线策略 | `YYMMDD-feature-name-overview.md` + `YYMMDD-feature-name-implementation.md` |
| 涉及长期演进、多阶段落地 | 多个完整命名文档，例如 `YYMMDD-feature-name-phase-1.md` / `YYMMDD-feature-name-phase-2.md` |

## 4. 写 spec 前要先收集的信息

写 spec 之前，不要直接开始写方案。应先了解上下文。

建议至少确认：

1. **用户问题 / 业务背景**
   - 用户遇到了什么问题？
   - 当前方案哪里不够好？
   - 这件事的价值是什么？

2. **当前系统现状**
   - 当前主链路是什么？
   - 入口函数 / API / 页面 / 任务在哪里？
   - 当前数据结构和协议是什么？
   - 当前 fallback、异常处理、日志和测试情况如何？

3. **已有能力**
   - 是否已有可复用模块？
   - 是否已有配置、工具函数、RPC、数据库表、缓存、prompt、组件？
   - 是否已有类似功能或历史方案？

4. **约束条件**
   - 是否需要兼容旧协议？
   - 是否要求灰度或 AB 实验？
   - 是否有性能、成本、稳定性或安全限制？
   - 是否有上线时间或回滚要求？

5. **验证方式**
   - 怎么判断方案有效？
   - 需要哪些单测、集成测试、手动验证和线上指标？

6. **待确认问题**
   - 哪些业务规则、展示文案、协议字段、上线范围或兼容策略还没确定？
   - 哪些方案取舍需要用户、产品、前端、后端或数据同学确认？
   - 哪些默认假设如果判断错了，会影响实现方向或上线风险？

写完 spec 后，如果仍有待确认问题，必须集中列出并向用户确认。只有当所有需要用户确认的问题都已经明确后，才应进入实现阶段。

## 5. 通用 spec 模板

下面模板可直接复制使用。

````markdown
# <功能 / 改造名称>方案概述

## 背景

说明为什么要做这件事。

建议包含：

1. 当前业务或技术场景
2. 现在遇到的问题
3. 为什么现有方案不足
4. 如果不做会有什么影响

## 当前现状

说明当前系统是怎么工作的。

建议包含：

1. 当前主链路
2. 关键入口
3. 关键数据结构 / 协议 / 配置
4. 当前 fallback / 异常处理
5. 当前问题点

可以使用流程图或文本调用链：

```text
入口
  -> 模块 A
  -> 模块 B
  -> 模块 C
  -> 输出
```

## 目标

本次要达成什么。

建议用编号列表：

1. 目标 1
2. 目标 2
3. 目标 3

目标要具体、可验证，避免写成泛泛的“优化体验”。

## 非目标

本次明确不做什么。

建议用编号列表：

1. 不做 A
2. 不改 B
3. 不影响 C

非目标非常重要，它能防止 scope 膨胀。

## 核心设计

说明总体方案和关键取舍。

建议包含：

1. 总体原则
2. 新旧方案对比
3. 为什么选择这个方案
4. 替代方案为什么暂时不选
5. 关键流程图

## 数据结构 / 协议设计

如果涉及结构体、接口、数据库表、缓存、消息、配置或 API，需要明确写出来。

示例：

```json
{
  "field": "value"
}
```

```go
type Example struct {
    Field string `json:"field"`
}
```

建议说明：

1. 字段含义
2. 是否必填
3. 默认值
4. 兼容旧数据的方式
5. 异常值处理方式

## 实现方案

说明具体要改哪些地方。

建议按模块拆分：

1. API 层
2. Service / Workflow 层
3. 数据访问层
4. RPC / 外部依赖
5. 配置
6. 日志 / 监控
7. 测试

每个模块建议写清楚：

- 改哪个文件 / 函数 / 类
- 新增什么结构或逻辑
- 如何兼容旧行为

## 兼容与降级

说明异常、失败、旧数据、旧客户端怎么处理。

建议包含：

1. 旧协议兼容
2. 旧数据兼容
3. 外部依赖失败时的 fallback
4. 灰度关闭后的行为
5. 回滚后是否会有脏数据或不兼容数据

## 可观测性

说明如何观察功能是否正常。

建议包含：

1. 关键日志
2. 关键指标
3. debug 信息
4. tracing / request id / session id
5. 告警条件

## 测试计划

说明如何验证。

建议至少包含：

1. 单元测试
2. 集成测试
3. 手动验证 case
4. 回归测试
5. 性能测试，如果涉及性能
6. AB 实验指标，如果涉及实验

## 上线与回滚

说明怎么上线，怎么回滚。

建议包含：

1. 是否需要配置开关
2. 是否需要灰度
3. 是否需要数据迁移
4. 回滚步骤
5. 回滚风险

## 建议实施顺序

按可落地顺序列出步骤：

1. 第一步
2. 第二步
3. 第三步

## 风险与待确认问题

列出仍不确定或需要评审的问题：

1. 风险 1
2. 风险 2
3. 待确认问题 1
````

## 6. 推荐写作结构

### 6.1 先写“为什么”，再写“怎么做”

推荐顺序：

```text
背景 -> 当前现状 -> 问题 -> 目标 -> 非目标 -> 核心设计 -> 实现细节 -> 验证
```

不要一上来就写代码改动。Reviewer 需要先理解为什么要改。

### 6.2 目标和非目标必须分开写

目标回答：这次要完成什么。

非目标回答：这次明确不做什么。

例如：

```markdown
## 目标

1. 支持新的搜索召回通道
2. 保持旧搜索链路可用
3. 支持灰度实验

## 非目标

1. 不替换旧搜索链路
2. 不修改前端协议
3. 不引入新的排序模型
```

这样可以有效避免需求边界失控。

### 6.3 当前现状要写得足够具体

不要只写：

```text
当前搜索逻辑比较复杂。
```

更推荐写：

```text
当前主链路为：

API Handler
  -> Workflow.Run
  -> QueryGenerator
  -> ParallelSearch
  -> ResultFusion
  -> ResponseWriter
```

并补充关键文件、函数、结构体或接口。

### 6.4 核心设计要讲取舍

不要只写最终方案，也要解释为什么这样做。

建议用表格比较：

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 方案 A | 改动小 | 能力有限 | 适合初版 |
| 方案 B | 能力完整 | 改动大 | 后续考虑 |

### 6.5 兼容和 fallback 必须写

任何工程方案都应说明失败时怎么办。

常见问题：

1. 新字段为空怎么办？
2. 老客户端不传字段怎么办？
3. 外部 RPC 超时怎么办？
4. 配置关闭后怎么办？
5. 灰度回滚后怎么办？
6. 旧数据反序列化是否会失败？

### 6.6 写完 spec 后必须确认待确认问题

写完 spec 后，不应默认直接进入实现。必须先检查文档中的“风险与待确认问题”，并判断其中哪些问题会影响实现方向、协议设计、展示效果、上线范围、兼容策略或回滚方式。

推荐流程：

```text
写完 spec
  -> 汇总所有待确认问题
  -> 主动向用户逐项确认
  -> 根据用户回复更新 spec 中的目标、非目标、方案和风险
  -> 确认没有剩余阻塞实现的问题
  -> 再进入实现
```

注意：

1. 如果待确认问题只是后续优化，不影响当前实现，可以明确标记为“非阻塞”。
2. 如果待确认问题会影响代码结构、接口字段、协议兼容、UI 文案、数据迁移或上线范围，则必须先问用户。
3. 用户确认后，应把确认结果同步回 spec，避免实现阶段和文档方案不一致。
4. 不要把重要假设藏在正文里，应集中列在“风险与待确认问题”或单独的“已确认决策”小节中。

## 7. 表达与引用原则

这部分只保留对 spec 有价值的写作约束，不展开基础 Markdown 规则。

1. 用清晰标题组织内容，用列表表达目标、非目标、步骤和规则。
2. 用表格表达方案对比、字段说明、实验分组、风险和测试 case。
3. 用代码块表达接口、结构体、配置、命令和流程。
4. 复杂链路可以用 Mermaid，但不要画得过大；节点 ID 建议使用英文或 camelCase，节点展示文本用双引号包裹。
5. 引用现有代码时，尽量写清楚文件路径、函数名、结构体名或字段名。
6. 描述当前行为前必须先确认代码现状，不要凭记忆写。
7. 不贴大段无关代码，只引用理解方案所需的关键结构、接口或函数签名。

## 8. 质量检查清单

写完 spec 后，可以用下面清单自查。

### 8.1 背景和目标

- [ ] 是否说明了为什么要做？
- [ ] 是否说明了当前问题是什么？
- [ ] 是否说明了本次目标？
- [ ] 目标是否具体、可验证？
- [ ] 是否写了非目标？

### 8.2 当前现状

- [ ] 是否说明了当前主链路？
- [ ] 是否列出了关键入口、模块或接口？
- [ ] 是否说明了当前数据结构或协议？
- [ ] 是否说明了当前 fallback 或异常处理？
- [ ] 是否说明了当前方案的不足？

### 8.3 方案设计

- [ ] 是否说明了核心设计原则？
- [ ] 是否说明了为什么选择这个方案？
- [ ] 是否比较了替代方案或关键取舍？
- [ ] 是否说明了对已有系统的影响范围？
- [ ] 是否明确哪些行为保持不变？

### 8.4 实现细节

- [ ] 是否列出了需要修改的模块？
- [ ] 是否说明了新增或修改的数据结构？
- [ ] 是否说明了接口、配置、缓存、数据库或消息协议变化？
- [ ] 是否说明了旧数据 / 旧客户端 / 旧配置兼容方式？
- [ ] 是否说明了错误处理和降级策略？

### 8.5 验证和上线

- [ ] 是否有单元测试计划？
- [ ] 是否有集成测试或手动验证 case？
- [ ] 是否有关键指标或日志？
- [ ] 是否说明了灰度方式？
- [ ] 是否说明了回滚方式？
- [ ] 是否列出了风险和待确认问题？
- [ ] 是否已向用户确认所有会影响实现的待确认问题？
- [ ] 用户确认后的决策是否已经同步回 spec？

## 9. 常见不推荐写法

避免以下写法：

1. 只写结论，不写背景和问题。
2. 只罗列代码改动，不解释设计取舍。
3. 没有非目标，导致需求边界不清。
4. 没有兼容、fallback、异常处理和回滚说明。
5. 没有测试计划，只写“开发后自测”。
6. 写成会议纪要，而不是可执行的技术方案。
7. 大段粘贴代码，缺少总结和判断。
8. 描述当前行为但没有确认真实代码现状。

## 10. 推荐落地流程

一个 spec 从提出到落地，推荐按下面流程推进：

```text
收集背景和现状
  -> 写 YYMMDD-feature-name-overview.md
  -> 汇总风险与待确认问题
  -> 向用户确认所有阻塞实现的问题
  -> 根据确认结果更新 spec
  -> 评审目标和非目标
  -> 必要时补充 YYMMDD-feature-name-implementation.md
  -> 评审实现方案
  -> 拆任务实现
  -> 按测试计划验证
  -> 灰度上线
  -> 记录结果和后续问题
```

如果需求较小，可以只写 `YYMMDD-feature-name-overview.md`，但仍建议至少包含：

1. 背景
2. 当前现状
3. 目标
4. 非目标
5. 核心设计
6. 测试计划

---

## 11. 最小可用 spec

如果时间很紧，至少写清楚以下内容：

```markdown
# <功能名称>方案

## 背景

为什么要做。

## 当前现状

当前怎么工作，问题在哪里。

## 目标

1. 目标 1
2. 目标 2

## 非目标

1. 不做 1
2. 不做 2

## 方案

准备怎么做，为什么这么做。

## 改动范围

1. 模块 A
2. 模块 B
3. 模块 C

## 兼容与降级

失败时怎么办，旧逻辑如何保持可用。

## 测试计划

如何验证。
```

这个最小版本也能保证 reviewer 理解：

- 为什么做
- 做什么
- 不做什么
- 怎么做
- 怎么验证

---

## 12. 总结

好的 spec 不追求长，而追求清楚。

最重要的是：

1. 先讲背景和现状，再讲方案。
2. 目标和非目标都要明确。
3. 关键取舍要写出来。
4. 数据结构、协议和改动范围要具体。
5. 兼容、降级、测试、上线和回滚不能缺。
6. 文档应该让 reviewer 不读完整代码，也能理解方案是否合理。

如果一份 spec 能让后续实现者、reviewer 和维护者快速理解“当时为什么这样设计”，它就是有价值的。
