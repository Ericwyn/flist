# Phase 6 详细设计：系统信息（磁盘用量）与弹窗历史守卫

> 对齐 `0.backend-design.md`、`1.开发阶段划分.md` 与 `5.存储驱动抽象设计.md`。承接 Phase 0 服务骨架、Phase 1 只读 / 下载、Phase 2 写 / 搜索、Phase 3 复制 / 收藏夹、Phase 4 分片上传、Phase 5 批量操作 / 打包下载。本文聚焦 Phase 6「系统信息」，并附带本阶段顺手修复的「弹窗下物理前进后退误切目录」交互问题。

## 1. 目标与边界

**目标**：补齐侧边栏底部的存储空间状态栏，让用户随时看到 root 所在文件系统的容量与剩余；同时修复弹窗打开时浏览器物理前进 / 后退会切换底层目录的交互错乱。

**本阶段包含**：

- **系统信息接口**：`GET /api/system/info`，返回 root 所在文件系统的磁盘总量 / 已用 / 剩余，复用既有 `storage.Usager` 可选接口。
- **侧边栏存储状态栏**：常驻底部、不随收藏夹滚动，显示总量 + 剩余 + 容量条；hover 容量条显示已用占比，占用 ≥ 90% 时容量条转红预警。
- **弹窗历史守卫（交互修复）**：弹窗打开时压入一枚「同址守卫」历史，物理前进 / 后退优先关闭顶层弹窗而非切换目录；守卫前后 URL 一致，消除地址栏闪跳，支持堆叠弹窗与各类关闭路径。

**本阶段不包含**：实时刷新 / 轮询磁盘用量（仅登录后加载一次）；配额 / 多挂载点的分项用量展示；最近访问列表。

**验收标准**：磁盘用量正确显示且驱动不支持时静默隐藏；弹窗打开状态下点击浏览器前进 / 后退仅关闭弹窗、底层目录与地址栏保持不变。

## 2. 关键设计取舍

| 取舍点 | 决策 | 说明 |
|--------|------|------|
| 用量来源 | **复用 `storage.Usager` 可选接口** | Phase 4 上传预检（`checkSpace`）已引入 `Usager`，Phase 6 直接复用，不新增 backend 接口。`local` 驱动经 `util.Usage`（statfs）取真实文件系统数据 |
| 驱动不支持时 | **静默隐藏，不报错** | 纯虚拟根 Mux 等未实现 `Usager` 时返回 `ErrNotSupported`；前端 `catch` 后置 `disk=null`，状态栏整体不渲染。系统信息属锦上添花，缺失不应打扰用户 |
| `used` 的计算 | **`total - free`（服务端算好）** | 不直接暴露原始 statfs 的 blocks，避免「可用 ≠ 空闲（root 预留）」带来的前端口径歧义；服务端统一算 `used = total - free`（并对 `total <= free` 做兜底取 0） |
| 刷新策略 | **登录后加载一次** | 容量变化缓慢，常态无需轮询；避免无谓请求。后续若需实时性可在写操作后失效缓存（本阶段不做） |
| 限流归类 | **只读，仅受全局限流（50/s）** | 不修改文件，不套写限流（10/s），与 `health` / `list` 等只读接口同档 |
| 弹窗前进后退冲突 | **守卫历史方案（pushState 同址守卫）** | 见 §7。相比「弹窗打开即禁用前进后退」更贴近移动端 / SPA 直觉（物理返回 = 关弹窗），且 URL 不跳动 |

## 4. 职责划分

延续「service 编排、backend 物理存取」原则：

| 留在 service / handler（后端无关） | 复用 backend（后端相关、物理） |
|---|---|
| `FileService.Usage`：类型断言探测 `Usager`，不支持时返回 `ErrNotSupported` | `Usage`：经 `resolve` 校验路径后 statfs 取真实容量 / 可用 |
| handler 计算 `used = total - free`（含 `total <= free` 兜底），组装 `SystemInfo` | — |

## 5. 系统信息接口

### 5.1 `GET /api/system/info`

```
// Response（成功）
{
  "disk_total": 512110190592,  // 文件系统总容量（字节）
  "disk_used":  321010892800,  // 已用 = total - free（字节）
  "disk_free":  191099297792   // 可用（字节）
}
```

- 需认证（位于 `protected` 路由组）。
- 始终查询 root（`/`）所在存储；当前单挂载场景下即整盘用量。
- 驱动不支持 `Usager`（纯虚拟根 Mux 等）→ 经 `failFileErr` 将 `storage.ErrNotSupported` 映射为 `4000`。

### 5.2 处理流程

1. handler 调 `files.Usage(ctx, "/")`：
   - service 对 `s.backend` 做 `storage.Usager` 类型断言，不支持 → `storage.ErrNotSupported`。
   - 支持 → 委托 `u.Usage(ctx, util.CleanAPIPath("/"))`，`local` 经 statfs 返回 `total` / `free`。
2. handler 计算 `used`：`total > free` 时取 `total - free`，否则取 0（防御异常 statfs 数据）。
3. 以 `model.SystemInfo` 经统一 `OK` 信封返回。

## 6. 错误码

**无新增错误码**。复用：`4000 bad_request`（驱动不支持 `Usager`，由 `ErrNotSupported` 映射）。前端对任意失败均静默隐藏状态栏。

## 7. 弹窗历史守卫（交互修复）

### 7.1 问题

弹窗（设置 / 重命名 / 确认删除等）浮在文件浏览器之上时，浏览器物理前进 / 后退会触发 `popstate`，被 `FileBrowser` 解释为「切换目录」——结果弹窗仍浮在上层，底层目录却被切走，地址栏也随之闪跳。期望行为应贴近移动端 / SPA 直觉：**物理返回 = 关闭最上层弹窗**，目录与地址栏不动。

### 7.2 弹窗栈：`lib/modalRegistry.ts`

模块级（不触发组件重渲染）维护当前打开的弹窗栈，`Modal` 组件在 `isOpen` 期间注册 / 卸载时移除：

| 导出 | 语义 |
|------|------|
| `nextModalId()` | 分配稳定弹窗 id（组件整生命周期复用） |
| `registerModal(id, close)` | 弹窗打开入栈；同 id 重复注册先移除旧项再入栈（置顶） |
| `unregisterModal(id, silent?)` | 弹窗关闭 / 卸载出栈；`silent` 供内部置顶复用不触发通知 |
| `openModalCount()` | 当前打开弹窗数（堆叠弹窗维护守卫用） |
| `closeTopModal()` | 调用最上层弹窗的 `onClose`；栈空返回 `false` |
| `subscribeModal(fn)` | 订阅栈变化（仅一个协调者：`FileBrowser`） |

`Modal` 把 `onClose` 存入 ref，使注册的 closer 始终调用最新处理函数（避免闭包陈旧）。

### 7.3 守卫不变式

**「只要有弹窗打开，当前历史位置之上即为一枚与当前 URL 相同的守卫条目。」**

`FileBrowser` 订阅弹窗栈变化协调：

- **栈 0 → >0**（首个弹窗打开）：`pushState` 一枚同 state、同 URL 的守卫（即便日后 forward 命中也只是同址 no-op）。
- **栈 >0 → 0**（弹窗经 X / 遮罩 / Esc / 按钮关闭，非物理返回）：仍停在守卫上，主动 `history.back()` 一格消费掉守卫，并置 `cleaningGuard` 吞掉随之而来的 `popstate`。

`popstate` 处理分三路：

1. `cleaningGuard` → 这是主动 `back()` 清理触发的，忽略并复位标志。
2. `guardActive` → 物理前进 / 后退消费了守卫：`closeTopModal()` 关顶层弹窗，不切目录；若仍有堆叠弹窗，`subscribeModal` 回调会立即补压新守卫，维持不变式。
3. 否则 → 正常目录历史恢复（`restore`）。

### 7.4 配套：`fsStore.restore` 同址短路

守卫复位后历史 state 的 `path` 可能与当前目录相同。`restore` 对此短路：仅校正 `historyIndex` 与 `replaceState` URL，**跳过重复列目录**，避免大目录无谓的网络刷新与列表闪烁。

```ts
restore: (path, index) => {
  if (path === get().currentPath) {            // 守卫复位 / 同址
    set({ historyIndex: index });
    window.history.replaceState({ index, path }, '', pathToUrl(path));
    return;
  }
  set({ historyIndex: index });
  get().navigate(path, false);
}
```

## 10. 前端展示约定

- **加载时机**：侧边栏挂载后 `api.system.info()` 加载一次；失败 `catch` 置 `null`。
- **渲染条件**：`disk && disk.total > 0` 才渲染；驱动不支持 / 异常时整块隐藏。
- **格式**：`formatBytes`（1024 进制，`B/KB/MB/.../PB`，默认 1 位小数）。
- **预警色**：`used/total >= 0.9` 容量条转 `rose`，否则 `blue`。
- **hover 提示**：容量条 `title` 显示「已用 X（N%）」。

## 11. 测试点

**后端**

- `service.FileService.Usage`：
  - backend 实现 `Usager`（`local`）→ 返回真实 `total`/`free`。
  - backend 不实现 `Usager`（纯虚拟根）→ `storage.ErrNotSupported`。
- `local.Usage`：正常路径返回非零 total；越界路径 → `ErrTraversal`（已有 `TestLocalUsage` / `TestLocalUsage_Traversal`）。
- handler `Info`：
  - 未登录 → 401。
  - 成功 → `disk_used == disk_total - disk_free`；`total <= free` 时 `used == 0`。
  - 驱动不支持 → `4000`。

**前端 / 端到端冒烟**

- 登录后侧边栏底部出现存储状态栏，数值与 `df` 量级一致；占用 ≥ 90% 显红。
- 驱动不支持用量时状态栏不出现，且无报错 toast。
- 弹窗守卫：打开任一弹窗 → 浏览器后退仅关弹窗、目录与地址栏不变；堆叠弹窗逐层后退关闭；经 X / Esc / 遮罩关闭后再后退能正常切目录（守卫已被清理）。

## 12. 与既有设计的一致性

- 系统信息复用既有 `storage.Usager`，**不新增 backend 接口**，service 编排、backend 物理存取分层不变。
- 错误词表延续 `storage` 统一错误 + handler 映射，`ErrNotSupported → 4000`，无新增错误码。
- 统一响应信封、限流、认证保护、SPA 静态托管均复用既有管道，无新增中间件。
- 弹窗守卫为纯前端历史协调，不触碰后端契约；`modalRegistry` 模块级状态不引入额外渲染开销。
