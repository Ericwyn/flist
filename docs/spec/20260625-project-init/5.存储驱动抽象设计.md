# flist 存储驱动抽象设计

> 本文档描述 flist 的存储后端（驱动）抽象，是对 `0.backend-design.md` 第 3 章「整体架构」的细化与演进。该抽象在 Phase 0/1/2 落地后引入，作为后续所有 Phase 的统一存储基座。已实现的本地能力行为不变，仅是分层重构 + 多挂载扩展点预留。

## 1. 背景与目标

flist 最初只支持本地文件系统。为支持后续扩展（WebDAV、特殊网盘等），且要支持「本地 + 多个网盘并存」，我们把「文件如何存取」从业务编排中抽离为 **`storage.Backend` 驱动接口**。

目标：

- **业务编排与存储实现解耦**：`FileService` 只做排序 / 分页 / 搜索匹配 / 预览嗅探 / 批量聚合等后端无关逻辑；寻址、路径安全、符号链接、权限语义等下沉到各驱动。
- **多后端并存**：本地 + 多个 WebDAV / 网盘以「挂载点」形式组合进同一个虚拟命名空间。
- **对外 API 零变化**：HTTP 接口、错误码、响应信封均不变；这是一次纯内部分层重构。
- **能力差异显式化**：用 `Caps` 声明驱动支持的能力，不支持的操作统一返回 `not_supported`。

## 2. 分层

```
Handler（HTTP / Range / 审计 / 错误码映射）
        │  只认 API 路径（/ 分隔）与 model 结构
        ▼
FileService（编排：排序/分页/搜索匹配/预览嗅探/批量聚合/能力校验）
        │  storage.Backend 接口（全链路 context.Context）
        ▼
┌──────────────┬──────────────┬──────────────┐
│ local（已实现）│ webdav（远期）│ 网盘（远期）  │
│ SafeResolve  │  HTTP/PROPFIND│  SDK/对象key │
│ symlink/perm │               │              │
└──────┬───────┴──────┬───────┴──────┬───────┘
       └──────────────┴──── storage.Mux（组合，自身亦是 Backend）
                 按挂载前缀路由 + 虚拟根
```

关键原则：**接口全部以「相对该后端根的 API 路径」为入参**，驱动内部负责翻译寻址并保证不逃逸自身命名空间。`SafeResolve` 等本地专属逻辑只存在于 `local` 驱动；`PathGuard` 中间件（拦 `..` / 绝对路径）作用于 API 路径层，保持通用。

## 3. 包结构

```
internal/storage/
├── backend.go        # Backend / File 接口、Caps、统一错误词表、可选接口（Walker/Usager/Uploader）
├── mux.go            # Mux：多挂载组合驱动（自身实现 Backend + Walker）
├── mux_test.go
└── local/
    └── local.go      # 本地文件系统驱动（吸收 SafeResolve/symlink/perm/hidden/fsop 调用）
```

`internal/util` 下的 `path.go` / `fsop.go` / `hidden_*` / `perm.go` / `name*` 保持不变，由 `local` 驱动复用。

## 4. 核心接口

```go
// 入参 p / src / dst 均为相对该后端根的 API 路径（/ 分隔、/ 开头）。
type Backend interface {
    Name() string
    Capabilities() Caps

    Stat(ctx, p) (*model.FileInfo, error)
    List(ctx, p, showHidden) ([]model.FileInfo, error)   // 已过滤隐藏，未排序未分页
    Open(ctx, p) (File, *model.FileInfo, error)          // 下载/预览，可随机读

    Mkdir(ctx, p) error
    Create(ctx, p) error                                 // touch
    Move(ctx, src, dst) error
    Remove(ctx, p) error
    Copy(ctx, src, dst) error
}

// File 必须可随机读以支撑 HTTP Range。
type File interface { io.ReadSeeker; io.Closer }

type Caps struct { Write, Copy, Upload, DiskUsage bool }
```

可选接口（驱动按需实现，service 用类型断言探测）：

```go
type Walker  interface { Walk(ctx, root, showHidden, func(relPath, info) error) error } // 高效递归遍历
type Usager  interface { Usage(ctx, p) (total, free uint64, err error) }                // 磁盘/配额（Phase 6）
type Uploader interface { /* Init / WriteChunk / Complete */ }                          // 分片上传（Phase 4）
```

### 4.1 统一错误词表

驱动把底层错误归一化为 `storage` 包的统一错误（`ErrNotFound` / `ErrTraversal` / `ErrForbidden` / `ErrExists` / `ErrInvalidName` / `ErrNotFile` / `ErrNotDir` / `ErrBadOp` / `ErrNotSupported`）。service 的 `ErrXxx` 别名指向它们，handler 用 `errors.Is` 映射对外错误码。三层共享同一组错误值，消除了原先 `service.ErrXxx` 与 `util.ErrPathTraversal` 两套错误的割裂。

## 5. 职责划分

| 留在 `FileService`（后端无关） | 进 `local` 驱动（后端相关） |
|---|---|
| 排序、分页 | `SafeResolve` / 路径穿越 / symlink 解析 |
| 搜索匹配（glob/子串）、超时/limit、截断聚合 | `os.*` 调用、`CopyPath/MovePath/RemovePath` |
| 预览文本嗅探 | 隐藏文件检测（平台分支） |
| 批量循环 + `OpResult` 聚合 | 权限格式化、跨分区 `EXDEV` 回退 |
| `Walker` 探测与基于 List 的兜底递归 | `FileInfo` 的 `IsSymlink/Unreachable` 填充 |

搜索的递归遍历：驱动实现 `Walker` 则用其高效实现；否则 service 退化为基于 `List` 的逐层递归（保证任何驱动都能搜索）。

## 6. 多挂载（Mux）

`Mux` 把多个 `Backend` 按挂载前缀组合成统一虚拟命名空间，**自身也实现 `Backend`**，因此上层无感：

```
/                → 虚拟根，List 返回各挂载点（呈现为一级目录）
/local/...       → 路由到名为 local 的挂载点，相对路径 /...
/box1/...        → 路由到名为 box1 的 WebDAV 挂载点
/box2/...        → 路由到名为 box2 的另一个 WebDAV 挂载点
```

规则：

- **虚拟根**：`List("/")` 返回各挂载点（合成目录条目）；不能在虚拟根或挂载点层级直接建目录 / 删除（`ErrBadOp`）。
- **路由**：取路径第一段为挂载点名，其余为相对该挂载点的路径；挂载点不存在 → `ErrNotFound`。
- **隔离**：各挂载点互不可见对方内容。
- **跨挂载 move/copy**：当前返回 `ErrNotSupported`，留待后续以「流式复制 + 删除」实现（不同后端间无法原子 rename）。
- **能力**：`Capabilities()` 取各挂载点能力的交集（保守对外）；单次操作的能力仍由命中的挂载点精确决定。
- **符号链接目标改写**：挂载点内驱动返回的 `SymlinkTarget`（相对其自身根）由 Mux 改写到虚拟命名空间（加挂载点前缀）。
- **搜索**：Mux 实现 `Walker`。虚拟根搜索时逐个挂载点遍历（relPath 前缀挂载点名）；挂载点子树搜索时委托对应后端。

### 6.1 装配（单一接入点）

`cmd/flist/main.go` 的 `buildBackend` 是唯一接入点：

- **当前默认**：单个 `local` 驱动「透明挂载」在根 —— 与改造前的 `--root` 行为完全一致，路径语义不变，Phase 0/1/2 验收标准不受影响。
- **引入远程驱动后**：在此构造各驱动并用 `storage.NewMux` 组合：

```go
mounts := []storage.Mount{
    {Name: "local", Backend: local.New(rootReal)},
    {Name: "box1",  Backend: webdav.New(cfg.WebDAV[0])},
    {Name: "box2",  Backend: webdav.New(cfg.WebDAV[1])},
}
return storage.NewMux(mounts), nil
```

> 多挂载启用后，对外 API 路径从「相对 root」变为「相对虚拟根」（即多一层挂载点前缀）。收藏夹存的相对路径需相应带挂载点前缀；前端面包屑天然适配。单挂载透明模式下无此前缀，向后兼容。

## 7. 下载 Range 注意事项（远程驱动关键约束）

`http.ServeContent` 要求 `File` 可 `Seek`。

- **本地驱动**：返回 `*os.File`，天然满足。
- **远程驱动**：需以「Range GET 支撑的惰性 seek reader」或「先落临时文件再 serve」实现可 Seek 语义。接口已用 `io.ReadSeeker` 形态的 `File` 为此留好口子，本地实现零成本。

这是实现 WebDAV / 网盘驱动时必须处理的首要事项。

## 8. 各 Phase 的落点

| Phase | 能力 | 落点 |
|---|---|---|
| 1 读 | List/Stat/Preview/Download | `Backend.List/Stat/Open` |
| 2 写+搜索 | Mkdir/Touch/Move/Delete/Search | `Backend.Mkdir/Create/Move/Remove` + `Walker` |
| 3 复制+收藏夹 | Copy | `Backend.Copy` + `Caps.Copy`；收藏夹路径含挂载点前缀（多挂载时） |
| 4 上传 | 分片上传 | 可选 `Uploader` 接口 + `Caps.Upload` |
| 5 批量+打包 | zip 流式打包 | `Walker` + `Open` 流式 zip，天然多驱动通用 |
| 6 系统信息 | 磁盘用量 | 可选 `Usager` 接口 + `Caps.DiskUsage`；多挂载时按挂载点分别展示 |
