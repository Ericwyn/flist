# flist 开发阶段划分

## 概述

将整体开发拆分为 7 个阶段（Phase 0–6），每个阶段完成后应具备可运行、可验证的完整功能。每阶段的具体接口字段、错误处理等实现细节，在进入该阶段时再单独细化设计；本文档只规划大致节奏与边界。

> 本文档与 `0.backend-design.md` 对齐。涉及的认证方式、上传协议、并发模型等以后端设计文档为准。

> **存储驱动抽象（贯穿各 Phase）**：Phase 0/1/2 落地后，文件存取已统一抽象为 `storage.Backend` 驱动接口（详见 `5.存储驱动抽象设计.md`）。`FileService` 只做后端无关的编排，本地能力收敛进 `storage/local` 驱动，并预留 `Mux` 多挂载组合（本地 + 多 WebDAV/网盘并存）。**这是一次对外行为不变的内部重构**，下列各 Phase 的接口与验收标准均不受影响；后续每个 Phase 的存储相关能力都落在对应的 `Backend` 方法或可选接口上（见各 Phase 末尾的「驱动落点」说明）。

> **前后端协同**：`frontend/` 已有前端原型模板。每个阶段同时完成前端 + 后端改动，阶段验收以「前端可操作、端到端可用」为准，而非仅后端接口可调通。

---

## 横切关注点（贯穿各阶段）

以下能力不单列阶段，而是随相关功能逐步落地：

- **限流中间件**：全局 + 写操作限流，在 Phase 0 搭建中间件管道时一并接入
- **审计日志**：写操作（创建/删除/移动/复制/上传）的审计记录，从 Phase 2 第一个写操作开始补齐
- **结构化日志**（`slog`）：Phase 0 起全程使用

---

## Phase 0：认证与服务骨架

**目标**：搭建可运行的服务端框架，完成用户认证体系与静态资源托管。

- HTTP 服务器初始化（chi 路由 + 中间件管道：Recovery / RequestID / Logger / RateLimit / CORS / Auth / PathGuard）
- 配置加载（启动参数 + 环境变量，`--root` 必填、缺失即启动失败）
- SQLite 初始化（用户表、会话表；开启 WAL、`busy_timeout`、写连接单连接）
- **会话令牌认证**（不透明 token，哈希入库）+ 登录/登出接口
- PathGuard 路径安全中间件（`SafeResolve` 兜底）
- 静态资源托管：`embed` 嵌入前端产物，含 SPA history 路由的 index.html fallback
- `/api/system/health` 健康检查

**验收标准**：服务启动后可通过 `/api/auth/login` 获取令牌；带令牌访问受保护接口成功、不带或失效令牌返回 401；前端静态页面可正常加载并刷新子路由不 404。

---

## Phase 1：文件浏览与下载（只读）

**目标**：打通完整的「读」路径，前端可浏览、预览、下载。

- `GET /api/fs/list` 目录列表（排序、隐藏文件控制、**分页**）
- `GET /api/fs/stat` 单文件/目录详细信息
- `GET /api/fs/preview` 预览（文本截断、图片）
- `GET /api/fs/download` 下载（`http.ServeContent`，**支持 Range / ETag**）
- 路径转换层（API 路径 <-> OS 路径），跨平台权限/隐藏文件展示
- 前端文件浏览器基础 UI

**验收标准**：前端可浏览目录树、分页加载大目录、查看预览、下载文件；视频/音频可拖拽进度（Range 生效）；跨平台路径正常。

> 说明：下载本属只读路径，与 preview 同源，故并入 Phase 1，使 Phase 1 成为完整的「读」里程碑；后续 Phase 4 专注「写」。

---

## Phase 2：基础文件操作

**目标**：实现文件和目录的创建、修改、删除等写操作。

- `POST /api/fs/mkdir` 创建目录
- `POST /api/fs/touch` 创建空文件
- `POST /api/fs/move` 移动 / 重命名（同分区 rename，跨分区降级复制+删除）
- `DELETE /api/fs/delete` 删除（递归，最大深度保护）
- `GET /api/fs/search` 搜索（串行遍历 + 超时 + 结果数量上限）
- 文件名校验（跨平台非法字符 + Windows 保留名）
- 写操作的审计日志（横切项起步）
- 前端右键菜单、重命名、删除确认弹窗

**验收标准**：可在前端完成新建、重命名、删除；搜索正常返回且大目录不拖垮服务；非法文件名被拦截。

---

## Phase 3：复制与收藏夹

**目标**：实现文件复制和目录收藏夹。可与 Phase 2 并行开发。

- `POST /api/fs/copy` 复制（递归，磁盘空间预检）
- 收藏夹 CRUD + 排序接口；收藏夹表（**存相对 root 的路径**）
- 前端复制粘贴交互（剪贴板状态管理）；剪贴板含「复制」与「剪切」两态，**剪切粘贴复用 Phase 2 已实现的 `/api/fs/move`**（Phase 2 前端仅暴露了同目录重命名，跨目录移动的 cut/paste 入口在此落地）
- 前端侧边栏收藏夹展示、失效检测与跳转

**验收标准**：可跨目录复制；可跨目录移动（剪切粘贴）；收藏夹支持增删、拖拽排序、点击跳转；root 迁移后收藏仍有效，失效项有标记。

---

## Phase 4：文件上传（分片）

**目标**：实现大文件友好的分片上传。

- 三段式分片上传：`POST /api/fs/upload/init` → `/chunk` → `/complete`
- 分片流式写入临时目录，不整文件入内存；合并后清理临时文件
- 上传前/合并前磁盘空间检查
- 并发写保护：**进程内路径级锁**
- 断点续传（重传缺失分片）
- 前端上传进度条、拖拽上传、断点续传交互

**验收标准**：可上传大文件且不阻塞服务；中断后可续传；并发上传同一目标不出现竞态损坏。

---

## Phase 5：批量操作与打包下载

**目标**：支持多文件同时操作和打包下载。依赖 Phase 1（下载）与 Phase 4（上传）。

- 批量上传（多文件并行，复用 Phase 4 分片能力）
- 批量删除、批量移动（复用 Phase 2 接口的数组入参）
- **打包下载**：服务端 zip 流式生成，选中多文件/文件夹一键下载
- 前端多选交互（Shift/Ctrl 点选、全选）

**验收标准**：可批量删除/下载；zip 流式生成、内容完整、不占用大量内存。

> 打包下载接口 `POST /api/fs/archive` 已在 `0.backend-design.md` 7.2 中定义。

---

## Phase 6：系统信息

**目标**：展示服务器端系统状态。仅依赖 Phase 0，可在 Phase 0 之后任意时机独立完成。

- `GET /api/system/info` 系统信息接口
- 磁盘空间检测（跨平台：Linux `Statfs` / Windows `GetDiskFreeSpaceEx`）
- 前端状态栏展示磁盘用量（总量、已用、剩余）

**验收标准**：前端状态栏正确显示磁盘空间，Linux / Windows 均可获取。

> 说明：磁盘空间检测工具与 Phase 3/4 的空间预检共用同一封装，谁先开发谁先落地。

---

## 阶段依赖关系

```
Phase 0 ──┬──> Phase 1 ──> Phase 2 ──┐
          │                          ├──> Phase 4 ──> Phase 5
          │           ┌──> Phase 3 ──┘                 │
          │           │   (与 Phase 2 并行)             │
          │           └─────────────────────────────────
          └──> Phase 6（仅依赖 Phase 0，可独立完成）
```

- **Phase 0** 是所有阶段的前置
- 主线：**Phase 1（读）→ Phase 2（写）→ Phase 4（上传）→ Phase 5（批量/打包）**
- **Phase 3**（复制+收藏夹）可与 Phase 2 并行
- **Phase 5** 依赖 Phase 1 的下载与 Phase 4 的上传
- **Phase 6** 仅依赖 Phase 0，可在任意时机插入
