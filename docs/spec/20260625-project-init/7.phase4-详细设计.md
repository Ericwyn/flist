# Phase 4 详细设计：文件上传（分片）

> 对齐 `0.backend-design.md`、`1.开发阶段划分.md` 与 `5.存储驱动抽象设计.md`。承接 Phase 0 服务骨架、Phase 1 只读、Phase 2 写 / 搜索、Phase 3 复制 / 收藏夹。本文聚焦 Phase 4「大文件友好的分片上传」。

> **实现状态（截至本次提交）**：Phase 4 后端 + 前端已全部实现并端到端验证通过（真实大文件分片上传 → 合并落盘 → 内容校验一致）。前端通过 `tsc --noEmit` 类型检查与 `vite build` 生产构建（产物已嵌入 `web/dist`），后端 `go test ./...` 全绿。本次提交在原设计之上另含 §15 的「`--debug` 调试日志」（输出上传分片 index/received/total_chunks，便于排查）。

## 1. 目标与边界

**目标**：实现三段式分片上传，大文件不整体入内存、不阻塞服务；支持断点续传与同目标并发写保护；前端拖拽上传 + 进度面板 + 续传交互。

**本阶段包含**：
- `POST /api/fs/upload/init`：初始化上传会话，返回 `upload_id` 与已收分片索引（断点续传）
- `POST /api/fs/upload/chunk`：上传单个分片（二进制 body，按 `index` 标识）
- `POST /api/fs/upload/complete`：校验分片齐全后合并落盘
- 分片流式写入 `<DATA_DIR>/uploads.tmp/<upload_id>/`，合并后清理
- 上传前（init）磁盘空间预检
- 并发写保护：进程内**路径级锁**（合并落点串行化）
- 断点续传：按**文件指纹**复用未完成会话，重传缺失分片
- 临时目录 24h 定时清理（孤儿分片兜底回收）
- `--max-upload` / `FLIST_MAX_UPLOAD` 配置项（设计文档 §5.1 已定义、代码此前未落地，本阶段补齐）
- 前端：拖拽上传、文件选择、并发分片上传（限并发）、进度面板、续传、覆盖确认

**本阶段不包含**：批量多文件并行编排的高级 UI、打包下载（Phase 5）；系统信息面板（Phase 6）。单文件上传链路打通即可，多文件以「逐文件复用本链路」的最简形式承载。

**验收标准**：可上传大文件且不阻塞服务；中断后可续传（同一服务生命周期内）；并发上传同一目标不出现竞态损坏；目标已存在时默认拒绝、前端可确认覆盖。

## 2. 关键设计取舍（已与用户确认）

| 取舍点 | 决策 | 说明 |
|--------|------|------|
| 分片暂存位置 / 合并 | **`<DATA_DIR>/uploads.tmp/<id>/` 暂存 + 流式拼接复制到目标** | 目录整洁、跨分区稳定、root 不被半成品污染；代价是合并时对大文件多一次完整读写 |
| 临时目录回收 | **24h 定时清理** | 超过 24h（按 mtime）的孤儿暂存目录由后台定时任务删除 |
| 断点续传状态 | **内存会话，重启丢弃** | 单进程少用户场景；服务重启后未完成上传需重传，孤儿分片由 24h 清理兜底 |
| 续传识别「同一文件」 | **后端按文件指纹复用会话** | init 携带 `fingerprint`（前端按 `name+size+lastModified` 计算）；命中未完成会话则返回其 `upload_id` 与已收分片，换浏览器 / 清缓存也能续（同一服务生命周期内） |
| 目标已存在 | **默认拒绝（`file_exists`）** | 上传覆盖风险大，由前端提示用户「覆盖」或「改名」后显式重试；与 copy/move 严格冲突语义一致 |

> **为何不持久化会话**：会话元数据（已收分片集合、指纹映射）随分片同生命周期，存内存最简单；重启场景在单进程 NAS 极少，重传成本可接受，孤儿分片有 24h 兜底，不残留。

## 4. 职责划分

延续 §5 存储驱动抽象的「service 编排、backend 物理存取」原则：

| 留在 service（后端无关、policy） | 进 local 驱动（后端相关、物理） |
|---|---|
| 上传会话注册表（内存，user 隔离） | 分片字节落盘（`uploads.tmp/<id>/<index>.part`） |
| 指纹复用、已收分片集合、会话过期 | 分片拼接合并到目标（`SafeResolve` + 落点不覆盖 / 覆盖语义） |
| 磁盘空间预检（`Usager`） | 暂存目录的创建 / 清理 / 24h sweep |
| 路径级锁串行化合并 | 文件名校验、父目录存在性 |
| 请求参数校验、`upload_id` 生成 | — |

**会话状态在 service（内存）、分片字节在 backend（磁盘）**，二者在同进程内同生命周期创建，天然一致；重启后会话 map 清空、分片文件成孤儿由 24h sweep 回收。`upload_id` 由 service 用随机 token 生成（同会话令牌的随机性），作为暂存目录名安全无注入。

## 5. 存储驱动接口（storage/backend.go 追加）

```go
// Uploader 是可选接口：驱动提供分片上传的物理存取（暂存 / 合并 / 清理）。
// 会话元数据（已收分片、指纹、user 归属、过期）由 service 的 UploadService 在内存维护，
// 驱动只负责按 upload_id 落盘分片字节并最终拼接到目标。upload_id 由 service 生成的随机
// token，驱动可安全地用作暂存目录名。
type Uploader interface {
    // StageChunk 将 upload_id 的第 index 个分片写入暂存区（先写临时文件再 rename，
    // 同 index 重传幂等覆盖）。返回写入字节数。暂存目录惰性创建。
    StageChunk(ctx context.Context, uploadID string, index int, r io.Reader) (int64, error)
    // MergeUpload 按序拼接 upload_id 的 [0..totalChunks-1] 分片到 dst，成功后删除暂存区。
    // dst 为相对 root 的 API 路径，驱动负责 SafeResolve + 文件名校验 + 父目录存在性。
    // overwrite=false 且 dst 已存在 → ErrExists；overwrite=true 则原子替换（先写同目录临时
    // 文件再 rename，避免合并中途损坏既有文件）。
    MergeUpload(ctx context.Context, uploadID, dst string, totalChunks int, overwrite bool) error
    // AbortUpload 删除 upload_id 的暂存区（幂等，不存在不报错）。
    AbortUpload(uploadID string) error
    // SweepStaging 删除 mtime 早于 now-maxAge 的孤儿暂存目录，返回清理数量。
    SweepStaging(maxAge time.Duration) (int, error)
}
```

- `local.New` 增加 `stagingDir` 入参（来自 `<DATA_DIR>/uploads.tmp`），构造时确保目录存在。
- **Mux**：分片上传只带 `upload_id` 无法仅按路径路由，跨挂载点上传留待远程驱动落地时实现（按 `upload_id → 挂载点` 路由）。当前单 `local` 透明挂载**不经 Mux**，service 直接对 `local` 做类型断言，故本阶段 Mux 暂不实现 `Uploader`。

## 6. 上传协议

### 6.1 `POST /api/fs/upload/init`

```json
// Request
{ "dir": "/documents", "name": "movie.mp4", "total_size": 1073741824,
  "chunk_size": 8388608, "fingerprint": "movie.mp4:1073741824:1700000000000" }
// Response
{ "code": 0, "data": {
    "upload_id": "<opaque>", "chunk_size": 8388608, "total_chunks": 128,
    "received": [0, 1, 2] } }   // 续传时返回已收分片索引，新会话为 []
```

**init 处理**：
1. 校验 `dir`（`SafeResolve` 经 backend、须为已存在目录）、`name`（文件名合法、非空）、`total_size>0`、`chunk_size`（落在 `[256KiB, 64MiB]`，越界归一到边界或拒绝）。
2. `total_size > max_upload`（配置 >0 时）→ `2009 upload_too_large`。
3. **磁盘预检**：`Usager.Usage(dir)` 取 free，`free < total_size` → `2005 disk_full`（驱动不支持 `Usager` 时跳过）。
4. 计算 `total_chunks = ceil(total_size / chunk_size)`。
5. **指纹复用**：在内存会话表按 `(userScope, cleanDir, name, fingerprint, total_size, chunk_size)` 查找未完成会话；命中则返回其 `upload_id` 与已收分片索引（续传），否则生成新 `upload_id`、建会话、返回空 `received`。

> **init 不因目标已存在而拒绝**：存在性的「默认拒绝覆盖」由前端预判（`fs.stat`）+ complete 的 `overwrite` 参数共同保证（见 §12.4）。若 init 在此拒绝，续传命中与覆盖重试都会卡住，故存在性检查只在 complete 做最终裁决。

### 6.2 `POST /api/fs/upload/chunk`

```
POST /api/fs/upload/chunk?upload_id=<id>&index=<n>
Content-Type: application/octet-stream
<二进制分片数据>
```

```json
// Response
{ "code": 0, "data": { "index": 5, "received": 6 } }  // received=已收分片总数（进度参考）
```

**chunk 处理**：
1. `upload_id` 查会话，不存在 / 已过期 → `2010 upload_not_found`。
2. `index` 校验 `0 <= index < total_chunks`，否则 `4000`。
3. body 经 `http.MaxBytesReader` 限制为 `chunk_size`（末片可更小），超限 → `4000`（或 413）。
4. 委托 `backend.StageChunk(upload_id, index, body)` 落盘（先 `<index>.part.tmp` 再 rename，幂等）。
5. 标记会话 `received[index]=true`、刷新 `lastActivity`。
6. **不经 `decodeJSON`**（那是 1MB JSON 通道），chunk 走独立的二进制读取路径。

> chunk 走全局限流（50/s）而非写限流（10/s）：分片数可能很多，10/s 会严重拖慢大文件上传。合并这一真正的「写」动作在 complete，complete 套写限流。

### 6.3 `POST /api/fs/upload/complete`

```json
// Request   { "upload_id": "<id>", "overwrite": false }
// Response  { "code": 0, "data": { "path": "/documents/movie.mp4" } }
```

**complete 处理**：
1. `upload_id` 查会话，不存在 → `2010 upload_not_found`。
2. 校验已收分片覆盖 `[0, total_chunks)`，缺片 → `2011 upload_incomplete`（data 附 `missing` 索引数组，便于前端补传）。
3. 取该会话目标路径的**路径级锁**（`util.PathLocker`），串行化同目标合并。
4. `overwrite=false` 且目标已存在 → `2004 file_exists`。
5. 委托 `backend.MergeUpload(upload_id, dst, total_chunks, overwrite)`：同目录临时文件拼接 → rename 到 `dst`（覆盖时先删旧再 rename）→ 删暂存区。
6. 成功后从会话表移除该会话；写审计日志（`upload`，path，结果）。

## 10. 并发与安全

- **路径级锁**（`util.PathLocker`）：按目标 `dst` 路径维护 keyed mutex，complete 合并时持锁，串行化同目标的并发完成，防止两个上传同时 rename 到同一路径损坏。分片写入不同 `<index>.part` 文件、各自 temp+rename 落盘，distinct index 天然无冲突可并行，不需锁。
- **路径安全**：`dir` 与合并落点经 backend 的 `SafeResolve` + `ValidateName`，不逃逸 root；`upload_id` 为随机 token，作暂存目录名无注入。
- **会话隔离**：会话表按 `userScope`（取自登录用户）隔离，指纹复用仅命中本人未完成会话。
- **限流**：init / complete 套写限流（10/s）；chunk 仅全局限流（50/s）。
- **body 限制**：chunk 按 `chunk_size` 限流式读取，不整体入内存；JSON 接口仍走 1MB `decodeJSON`。
- **磁盘**：init 预检目标文件系统 free ≥ total_size；暂存与合并的瞬时双倍占用在同盘场景由预检 + 合并后即时清理缓解（文档提示运维 DATA 目录与 root 同盘时留足空间）。

## 11. 临时目录清理

- **即时**：complete 成功后删本会话暂存区；abort（前端取消）调用 `AbortUpload` 删暂存区。
- **定时**：`main.go` 后台 goroutine（复用或并列于会话清理 ticker）周期调用：
  - `UploadService.SweepSessions()`：移除内存中 `lastActivity` 早于 24h 的会话。
  - `backend.SweepStaging(24h)`：删 `uploads.tmp` 下 mtime 早于 24h 的孤儿目录。
- 数据目录结构补充：`<DATA_DIR>/uploads.tmp/<upload_id>/<index>.part`。

## 12. 前端

### 12.1 数据层
- `lib/api.ts`：
  - `fs.uploadInit(dir, file)`：计算 `fingerprint = name:size:lastModified`，POST init，返回 `{upload_id, chunkSize, totalChunks, received}`。
  - `fs.uploadChunk(uploadId, index, blob)`：`POST .../chunk?upload_id&index`，body 为 `Blob`，`Content-Type: application/octet-stream`。
  - `fs.uploadComplete(uploadId, overwrite)`：POST complete。
- `types.ts`：`UploadTask = { id, file, dir, status, progress, uploadId?, error? }`（status: `pending|uploading|paused|done|error|conflict`）。

### 12.2 上传队列（uploadStore.ts）
- 维护任务队列；每个文件：init → 并发上传缺失分片（并发上限如 3）→ complete。
- 进度 = 已收分片数 / 总分片数；续传时 init 返回的 `received` 直接跳过。
- 目标已存在：入队前经 `fs.stat` 预判，存在则任务转 `conflict` 态（不自动上传），由用户选择覆盖 / 改名 / 取消，详见 §12.4。覆盖语义在 complete 传 `overwrite=true`，init 始终成功创建会话、不因存在性拒绝。
- 完成 / 失败后刷新当前目录（仅当上传目标为当前目录时）。

### 12.3 UI
- **拖拽上传**：`FileBrowser` 文件区作为放置区，拖入文件 → 加入队列上传到 `currentPath`；高亮放置态。
- **工具栏「上传」按钮**：触发隐藏 `<input type=file multiple>`。
- **`UploadPanel`**：固定底部，列出任务名 / 进度条 / 状态 / 取消按钮；全部完成可收起。
- **覆盖确认**：冲突任务行内提供「覆盖」「改名后重试」「取消」。

### 12.4 冲突处理细化（覆盖语义落地）

为兼顾「默认拒绝覆盖」与「用户可显式覆盖」：
- 前端入队时先 `fs.stat(dir/name)` 预判是否存在；存在则该任务直接置 `conflict` 态，不自动上传，由用户选择：
  - **覆盖**：以 `overwrite=true` 走完整 init→chunk→complete（complete 传 `overwrite=true`）。
  - **改名**：用户输入新名，以新名重新入队。
  - **取消**：移除任务。
- 后端 complete 仍以 `overwrite` 参数为准做最终裁决（TOCTOU 兜底）：`overwrite=false` 且存在 → `file_exists`。

> init 自身**不**因目标已存在而拒绝（仅校验 dir/name/size/disk）；存在性的「默认拒绝」由前端预判 + complete 的 `overwrite` 共同保证。这样续传命中、覆盖重试都不会卡在 init。

## 13. 测试点

**后端**
- `util.PathLocker`：同 key 串行、不同 key 并行；Unlock 后可再 Lock。
- `local` Uploader：StageChunk 落盘（含同 index 幂等重传）、MergeUpload 顺序拼接内容正确、缺片报错、overwrite=false 落点存在报 ErrExists、overwrite=true 原子替换、AbortUpload 清理、SweepStaging 按 mtime 清孤儿。
- `service.UploadService`：init 新建会话 / 指纹命中续传返回 received、chunk 标记 received、complete 缺片 `upload_incomplete` 附 missing、complete 成功内容完整、磁盘预检不足 `disk_full`、超 max_upload `upload_too_large`、user 隔离（他人指纹不命中）、SweepSessions 过期移除。
- `handler` e2e：init→分片乱序上传→complete→list 出现文件且内容字节一致；中断（部分分片）后重新 init 返回 received、补传剩余 chunk→complete 成功；覆盖（overwrite）替换内容；并发两个 upload 完成到同路径其一 file_exists。

**端到端冒烟**：登录 → 拖拽大文件上传 → 进度增长 → 完成出现在列表 → 模拟中断后续传 → 上传同名触发冲突 → 覆盖成功。

## 14. 与既有设计的一致性

- 复用 §5 存储驱动抽象的可选接口模式（`Uploader` 与 `Walker`/`Usager` 同构），service 经类型断言探测、不支持则 `not_supported`。
- 错误词表延续 `storage` 包统一错误 + handler `errors.Is` 映射。
- 审计、限流、路径安全、统一信封均复用既有管道，无新增中间件。
- 配置 `--max-upload` 补齐 `0.backend-design.md` §5.1 既定项，优先级遵循「参数 > 环境变量 > 默认」。

## 15. 调试日志（本次提交追加）

> 上线后排查上传问题时，访问日志只记响应字节数（chunk 接口固定返回约 60+ 字节的 JSON 信封，看不出分片信息）。为便于确认「是否分片、传了几片、是否传齐」，新增 `--debug` 开关输出上传细节。**默认关闭，不影响正常运行的日志噪音。**

- **配置**：`--debug` / `FLIST_DEBUG=true`（见 `0.backend-design.md` §5.1）。开启后日志级别由 Info 降到 Debug。
- **实现**：`main.go` 用 `slog.LevelVar` 持有日志级别，配置加载后按 `cfg.Debug` 动态 `Set(slog.LevelDebug)`；handler 在上传三接口输出 Debug 日志。
- **输出字段**：
  - `upload init`：`upload_id` / `dir` / `name` / `total_size` / `chunk_size` / `total_chunks` / `already_received`（续传时已有分片数）
  - `upload chunk`：`upload_id` / `index`（本次分片序号）/ `received`（累计已收不同分片数）
  - `upload complete`：`upload_id` / `path`（落盘路径）/ `overwrite`；缺片时为 `upload complete incomplete` + `missing` 索引数组
- **判读要点**：前端默认 3 个分片并发上传，分片**乱序到达**，故 `received` 不严格等于 `index+1`（并发在途分片未计入属正常）。确认成功只看两点：出现 `upload complete` 且 `received` 最终达到 `total_chunks`；落盘文件大小与原文件一致。
