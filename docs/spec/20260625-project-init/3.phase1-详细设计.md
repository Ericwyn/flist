# Phase 1 详细设计：文件浏览与下载（只读）

> 对齐 `0.backend-design.md` 与 `1.开发阶段划分.md`，承接 Phase 0 的服务骨架与 `SafeResolve`。本文聚焦 Phase 1「读」路径的可落地细节，不涉及任何写操作（mkdir/touch/move/delete/copy/upload 属 Phase 2+）。

## 1. 目标与边界

> **存储驱动对齐（架构更新）**：本阶段的读路径已重构为面向 `storage.Backend` 驱动接口（详见 `5.存储驱动抽象设计.md`）。下文描述的 `FileService.List/Stat/PreviewText/OpenForDownload` 仍是对外契约，但内部已拆分为「service 编排 + 驱动存取」：
> - **驱动落点**：`List → Backend.List`、`Stat → Backend.Stat`、`PreviewText/OpenForDownload → Backend.Open`；排序/分页/文本嗅探留在 service。
> - **本地实现**：`SafeResolve`、符号链接解析、权限格式化、隐藏文件检测均在 `storage/local` 驱动内，行为与下文完全一致。
> - **方法签名**：各方法已统一加 `context.Context` 首参（远程驱动需要取消/超时；本地驱动忽略）。
> - 对外 HTTP 接口、字段、错误码不变，本节其余内容仍然有效。

**目标**：打通完整的「读」路径，前端可浏览目录、查看预览、下载文件（含 Range）。

**本阶段包含**：
- `GET /api/fs/list`：目录列表（排序、隐藏文件控制、分页）
- `GET /api/fs/stat`：单文件/目录详细信息
- `GET /api/fs/preview`：文本预览（截断）
- `GET /api/fs/download`：下载/内联（`http.ServeContent`，支持 Range / ETag）
- 路径转换层落地（API 路径 ↔ OS 路径，已在 Phase 0 的 `util` 起步）
- 跨平台权限格式化、隐藏文件检测、符号链接展示
- 前端文件浏览器从 mock 切换为真实 API（只读）

**本阶段不包含**：写操作、搜索（Phase 2）、收藏夹（Phase 3）、上传（Phase 4）、打包下载与多选批量（Phase 5）、系统信息（Phase 6）。文本编辑保存属写操作，故 Phase 1 预览为**只读查看**。

**验收标准**：前端可浏览目录树、分页加载大目录、查看预览、下载文件；视频/音频可拖拽进度（Range 生效）；跨平台路径正常。

## 3. 数据模型

```go
// model/file.go
type FileInfo struct {
    Name        string    `json:"name"`
    Type        string    `json:"type"`          // "file" | "dir"
    Size        int64     `json:"size"`          // 目录为 0
    Mode        string    `json:"mode"`          // 八进制字符串，如 "0644"
    ModTime     time.Time `json:"mod_time"`
    IsSymlink   bool      `json:"is_symlink"`
    SymlinkTarget string  `json:"symlink_target,omitempty"` // 尽力解析，相对 root 的 API 路径
    Unreachable bool      `json:"unreachable,omitempty"`     // 符号链接目标越界/不可达
}

type ListResult struct {
    Path     string     `json:"path"`       // 规范化后的 API 路径
    Total    int        `json:"total"`      // 过滤后的总条目数（分页前）
    Page     int        `json:"page"`
    PageSize int        `json:"page_size"`
    Items    []FileInfo `json:"items"`
}
```

> `os` 字段（平台标识）由 `/api/system/info`（Phase 6）提供；Phase 1 前端对 `mode` 仅做展示，不深度解读。

## 4. 接口设计

统一响应信封同 Phase 0：`{ "code", "message", "data" }`。所有 `fs` 接口需认证，经 `Auth → PathGuard` 管道。

### 4.1 GET /api/fs/list

```
Query:
  path       相对 root 的 API 路径，默认 "/"
  sort       name | size | mtime，默认 name
  order      asc | desc，默认 asc
  show_hidden  true | false，默认 false
  page       从 1 起，默认 1
  page_size  默认 200，最大 1000
```

排序规则：**目录优先**，目录组与文件组各自按 `sort` 键排序；`name` 排序大小写不敏感。分页在排序与过滤之后切片。

```json
{ "code":0, "data": {
  "path":"/documents", "total":1532, "page":1, "page_size":200,
  "items":[ { "name":"a.txt","type":"file","size":1024,"mode":"0644",
              "mod_time":"2026-06-20T10:00:00Z","is_symlink":false } ]
}}
```

### 4.2 GET /api/fs/stat

```
Query: path
```
返回单个 `FileInfo`。目标不存在 → `2001 path_not_found`。

### 4.3 GET /api/fs/preview

```
Query: path
```
仅对**文本文件**返回内容预览（其余类型由前端改用 download 内联加载）：

```json
{ "code":0, "data": {
  "type":"text", "content":"前 N 字节…", "truncated":true,
  "size":20480, "preview_bytes":65536
}}
```

判定与限制：
- 文本判定：先看扩展名白名单，再对前 512 字节做二进制嗅探（含 NUL 字节即视为二进制）
- 截断上限：默认 **64 KiB**，超出 `truncated=true`
- 非文本：返回 `{ "type":"binary" }`（或 `image`/`video`/`audio` 提示），`content` 为空，前端转用 download 内联

### 4.4 GET /api/fs/download

```
Query:
  path
  download   可选，=1 时强制附件下载（Content-Disposition: attachment）
```
- 用 `http.ServeContent` 提供，自动支持 **Range / If-Range / Last-Modified**
- 额外设置 **ETag**：由 `size + mod_time` 派生（`"<size>-<modUnixNano>"`），支持 `If-None-Match`
- `Content-Type` 由扩展名 + 内容嗅探得出
- 默认**内联**（便于 `<img>`/`<video>`/`<audio>` 直接加载并拖拽进度）；带 `download=1` 时附件下载
- 目标为目录 → `2007 not_a_file`（Phase 1 不打包，打包属 Phase 5）

> **媒体鉴权**：`<img>/<video>` 无法携带 `Authorization` 头。Phase 0 登录已写入 `HttpOnly` Cookie `flist_session`，且 `Auth` 中间件支持从 Cookie 取令牌，故媒体元素直接以 `/api/fs/download?path=…` 为 `src` 即可凭 Cookie 鉴权。

## 5. 服务层逻辑（service/file.go）

`FileService` 持有启动时缓存的 `rootReal`。所有方法入口先 `SafeResolve(rootReal, apiPath)`。

**List**
1. `SafeResolve` → 本地目录路径；非目录 → 错误
2. `os.ReadDir` 读取条目
3. 逐条 `Lstat`：判断是否符号链接；是则 `os.Readlink` + 尝试 `SafeResolve` 目标，越界则 `Unreachable=true`、不展示 target
4. 过滤隐藏文件（`show_hidden=false` 时，按平台 `IsHidden` 剔除）
5. 排序（目录优先 + sort 键 + order）
6. 记 `Total`，再按 `page/page_size` 切片
7. 组装 `FileInfo`（`Mode` 经 `util.FormatMode`）

**Stat**：`SafeResolve` → `Lstat` → 单个 `FileInfo`。

**PreviewText**：`SafeResolve` → 必须普通文件 → 读前 `previewBytes` → 文本嗅探 → 返回内容或 binary 提示。

**OpenForDownload**：`SafeResolve` → 必须普通文件 → 返回 `*os.File` 与 `os.FileInfo`，由 handler 交给 `http.ServeContent`（handler 负责 `Close`）。

> 所有路径错误统一映射：`ErrPathTraversal → 2002`；`os.ErrNotExist → 2001`；`os.ErrPermission → 2003`；目录当文件用 → `2007`。

## 6. 跨平台封装（util）

- `perm.go`：`FormatMode(os.FileMode) string`，取低 9 位格式化为 4 位八进制（如 `0644`）
- `hidden_linux.go` / `hidden_windows.go`：`IsHidden(name string, info os.FileInfo) bool`
  - Linux：`strings.HasPrefix(name, ".")`
  - Windows：读取 `FILE_ATTRIBUTE_HIDDEN`（`syscall.Win32FileAttributeData`），失败降级为 `.` 前缀判断
- `filekind.go`：`IsTextExt`/`SniffText`、`DetectKind`（folder/text/image/video/audio/unknown，供前端图标与预览决策；后端 `type` 仍只返回 file/dir，kind 由前端按扩展名推导，后端不强加）

> Phase 1 在 Linux 开发与验证；Windows 的 `hidden_windows.go` 同步实现，靠 build tag 隔离，CI 矩阵编译保证可构建。

## 7. 错误码补充（在 Phase 0 基础上）

| Code | Message | 说明 |
|------|---------|------|
| 2001 | path_not_found | 路径不存在 |
| 2002 | path_traversal | 路径越界 |
| 2003 | permission_denied | 文件权限不足 |
| 2007 | not_a_file | 期望文件却为目录（download/preview） |
| 2008 | not_a_dir | 期望目录却为文件（list） |

## 8. 前端改造

Phase 0 的文件浏览器是基于 `id`/`parentId` 的内存 mock，与后端基于 path 的模型不兼容，Phase 1 整体切换为真实 API。

**数据层**
- `types.ts`：`FileEntry { name, type:'file'|'dir', size, mode, modTime, isSymlink, ... }`，浏览以 **path 字符串**为中心（不再有 id/parentId）
- `lib/api.ts` 追加：`fs.list(path, {sort,order,showHidden,page,pageSize})`、`fs.stat(path)`、`fs.preview(path)`，以及 `fs.downloadUrl(path, {download})` 构造内联/下载 URL
- `fsStore.ts`：保存 `currentPath`、`entries`、分页、排序、加载态、错误；`navigate(path)` 拉取 `list`；面包屑由 path 切分得到；前进/后退用路径历史栈

**UI**
- `FileBrowser`：双击目录 → `navigate`；双击文件 → 文本走 `preview`、媒体走 download 内联预览；面包屑、排序切换、分页/加载更多
- `PreviewModal`：文本只读展示（Phase 1 不保存）；图片 `<img src=downloadUrl(inline)>`；音视频 `<video>/<audio>` 同源 Cookie 直连，支持拖拽（Range）
- `FileIcon`：按扩展名推导 kind 选图标
- `Sidebar`：磁盘用量与收藏夹在 Phase 1 暂以占位/隐藏处理（分别属 Phase 6 / Phase 3），保留登出与用户信息
- 401 处理：API 客户端遇 `1001` 触发 `authStore` 登出，回到登录页

## 9. 测试点

**后端**
- `util.FormatMode`：常见权限位格式化
- `util.IsHidden`（linux）：`.` 前缀
- `util.SniffText`：文本/含 NUL 二进制
- `service.List`：排序（name/size/mtime + asc/desc）、目录优先、隐藏过滤、分页切片、`Total` 正确、符号链接标记、越界目标 `Unreachable`、非目录报错
- `service.Stat`：文件/目录/不存在
- `service.PreviewText`：文本截断 `truncated`、二进制识别、目录报错
- `handler.download`（httptest）：完整下载、`Range: bytes=` 返回 206 与正确分片、`If-None-Match` 命中 304、目录报错、越界 403/2002
- `handler.list/stat/preview`：参数解析、错误码映射、需认证

**端到端冒烟**：登录 → 列目录 → stat → 文本预览 → 图片内联 → Range 下载分片。

## 10. 安全要点

- 所有 `fs` 接口入口 `SafeResolve`，前缀检查在符号链接解析之后（复用 Phase 0 实现）
- 符号链接目标越界仅标记 `Unreachable`，不解析、不允许后续访问
- `download`/`preview` 仅接受普通文件，拒绝目录与特殊文件
- 列目录对单条 `Lstat`/`Readlink` 失败降级（记录并跳过或标记），不因个别坏条目中断整目录
- 系统关键路径黑名单（`/proc`、`/sys` 等）按 `0.backend-design.md` 9.4，Phase 1 若 root 落在敏感区由部署约束，黑名单细化可在写操作阶段强化
